# PROMPT 0003 — verbatim

> since the introduction of project as a home, aka workspace scopes and global
> scopes, i feel like one potential branch that we might want to explore is that
> once we start launching more and more agents, or agent teams or agent factories
> we might want to have agent or identity attribution, and agent memory as an
> optional choice, not mandatory or forced, but modular and configurable.
> Oooooooooohhh.... i just realized another really cool idea, btw this is one of
> our tenets, i love when backlog-mcp or a novel product just bolts on top of the
> existing world, thats one of our tenets, of course bolting on top doesn't mean
> to limit our visionary goals, we shouldn't handicap achieve our vision just
> because we can't bolt on top of something, but if we can bolt on top of
> something, then life is great. For example what if we integrate with worktree
> natively, for example during agentic delegation i heavily use worktrees, as a
> result agents might end up having decoupled access to the source of truth,
> because they are working in branches, which is useful for the work decoupling
> but what if we make changes to the agent instructions or the vision or the
> tenets, or we write a requirements intake document, just the ergonomics to bolt
> on top of the worktrees or to work seamlessly with worktrees will make the
> backlog-mcp to have an architectural and ergomomical moats. I don't fully know
> how this will play out, but I know we need to do something about it, like
> backlog-mcp seamlessly understands that there are multiple agents potentially
> working in different worktrees and if lets say the main brain writes something
> in the main repo, the agent running in the worktree could potentially have
> access to the main repo artifacts if needed, otherwise it is scoped. I feel
> like i am scraping several things that needs to be explored further and
> distilled as vision, approaches and proposals, like, agent identities, agent or
> identities potentially having their own memory and having the access to the
> worktree, workspace and global level memories, maybe as a user i can have my
> own memory as well, that agents don't need to know? sometimes isolation is
> needed, sometimes not, this is all about the architecture and the ergonomics.
> The bolting tenet is crucial because all of this should feel very native, like
> all of this might already exist and we just bolt on top, like even if you
> haven't use backlog-mcp and you have been accumulating all these memories,
> skill files, context files and what not, you can still start using the
> backlog-mcp and the ergonomics of it will make so much sense that you are not
> going to look back any more. These are my rough ideas that i want you to
> iterate on and write into artifacts, after research or whatever it takes to
> shape these ideas into the real world proposals and judgment calls. Are these
> ideas aligned to our trunk, what technologies do they require and so on.
