# Architecture Decision Records

This directory contains Architecture Decision Records (ADRs) for backlog-mcp.

## What is an ADR?

An ADR documents an important architectural decision along with its context and consequences. It helps future contributors understand why certain choices were made.

- [0118. Proactive Recall — Hook-Driven Memory Injection at the Client Boundary](./0118-proactive-recall-hooks.md) **[Proposed; design-first; continues memory thread 0092.5]** - Agents skip `backlog_recall` (5+ times) because recall is a *voluntary tool call* competing with model confidence — a will-it-fire problem no louder instruction fixes. The field's answer (Mem0/MemPalace/claude-mem) isn't a smart gate: they fire recall on **every** `userPromptSubmit`, unconditionally, made viable by (1) low latency and (2) injecting **stubs not bodies**. Our blocker is purely latency — `npx backlog-mcp recall` pays 2–5 s cold start per turn. Decision: move recall from the prompt layer to a kiro-cli **hook** (`agentSpawn`→`wakeup` once; `userPromptSubmit`→per-turn stub injection), and add a warm-daemon `GET /memory/recall` route (delegating to existing `core/recall.ts`, `full=false`) so the hook is a <50 ms `curl`, not a Node spawn. No intent classifier; stubs-only + score floor + fail-open. Extends the ADR 0092.5 two-stage recall contract from *what* recall returns to *when* it fires - 2026-06-20
- [0110. Vite for Viewer Dev (HMR) — Static Bundle in Prod](./0110-viewer-dev-vite-prod-static.md) **[Proposed; design-first]** - Live testing of the esbuild dev-HMR client (nisli ADR 0021) exposed a structural defect: re-importing the bundled entry to apply a change instantiates a **second copy of the entire `@nisli/core` runtime**, so live elements and new setups disagree about lifecycle/reactive context (`onCleanup() called outside setup()`). Root cause is intrinsic to bundled-entry re-import: esbuild scope-hoists modules and offers no module runtime (esbuild #1940/#464/#802 — declined), so granular HMR is impossible without hand-rolling Vite's dev core. The operator retired the "prod-shaped bundle in dev" constraint. Decision: **Vite for dev (unbundled ESM + `import.meta.hot`), Rollup static hashed bundle for prod** — Vite is build/dev-time only (a viewer `devDependency`), never in the published package; prod runtime + ADR 0108 cache posture unchanged. Supersedes the esbuild-HMR approach (0.49.x) for the viewer - 2026-06-19
- [0108. Content-Hashed Viewer Assets — Cache-Busting for Zero-Stale Releases](./0108-viewer-asset-cache-busting.md) **[Proposed; design-first]** - A published viewer fix didn't reach users on a normal refresh: entry assets (`main.js`/`main.css`) are served at stable, unhashed URLs with no `Cache-Control`/`ETag` (verified live via `curl`), so browsers heuristically cache a stale `/main.js` across restarts. Fix = content-hash all build outputs (esbuild `entryNames: '[name]-[hash]'`) + rewrite `index.html` from the metafile; serve hashed assets `immutable` and the 357-byte `index.html` `no-cache`. Only the tiny HTML is ever revalidated; the 1 MB bundle is fetched once per actual change. Surfaced by shipping ADR 0008.1 (nisli) - 2026-06-18

## Format

Each ADR follows this structure:
- **Context**: What problem are we solving?
- **Proposed Solutions**: What options did we consider?
- **Decision**: What did we choose and why?
- **Consequences**: What are the implications?
- **Implementation**: How do we build it?
