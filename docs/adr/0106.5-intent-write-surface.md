# 0106.5 Intent Write Surface — Substrate-Declarations, One Core Executor, No Generic MCP CRUD

**Date**: 2026-07-16
**Status**: Proposed
**Backlog Item**: TASK-0688
**Thread**: implements ADR 0106 Phase 2; continues ADRs 0106.1–0106.4
**Informs / is informed by**: ADR 0094 (write attribution and journal) · ADR
0104 (local-first deployment posture) · ADR 0113 R7 (substrate-declared
semantic intents) · ADR 0114 (progressive disclosure and no-legacy migration)

## Decision summary

The MCP write boundary will expose a small set of substrate-declared semantic
verbs and will delete the generic `backlog_create` and `backlog_update` tools in
the same change.

The shared substrate compiler owns declarations, validation, tool-name
selection, and an opaque compiled DTO. The server owns the consumer port, one
transport-free intent executor, and the generated MCP adapters. Every create
intent converges on one core `createEntity` operation; transitions and
relations converge on the existing core update/storage primitives.

There is no guessed CRUD and no project-supplied callback. The initial set is
the sixteen real intents ratified for this phase with the ADR 0113 owner.
Memory keeps its existing `backlog_remember` and `backlog_forget` tools and is
not dual-registered.

The low-level tail escape hatch remains the local CLI's generic
`backlog create` / `backlog update` commands. It is not a second MCP contract.
Docs-native substrates also remain directly editable as human-visible files.

## The founder instinct still stands

The original substrate design was explicitly anti-tool-bloat:

> “I didn't want 15 tools … creation/update tells you the exact shape of the
> substrate so the agent sees full shapes only when it needs to.”

That instinct, recorded in the product prompts
([prompts 5 and 12](https://gkoreli.com/one-hundred-pull-requests/prompts)),
was about **lazy context**, not loyalty to generic CRUD. The agent should not
pay for every substrate shape before it needs one.

ADR 0106 later chose intent verbs because the delivery mechanism changed:
modern harnesses can defer full tool schemas and load only a tool's name and
discovery description until selection. A narrow verb therefore preserves the
founder's original progressive-disclosure rule:

1. the unloaded surface carries only a crisp semantic name and description;
2. selecting the verb reveals only the inputs for that intent;
3. the complete substrate model remains behind the port;
4. uncommon writes fall back to an explicit low-level local path instead of
   forcing every possible mutation into the MCP manifest.

This does **not** license a tool per operation or a generated CRUD matrix.
Substrates declare only verbs with a real domain meaning. The compiler never
infers `create_<type>`, `update_<type>`, or status verbs from a schema. A new
verb requires an explicit declaration and product ruling; schema existence
alone never creates one.

This reconciles both halves of the founder's direction: semantic language at
the agent boundary, and full shapes disclosed only at the moment of need.

## As-built audit

### The public write surface still speaks storage

`packages/server/src/tools/backlog-create.ts:9-42` registers
`backlog_create`. Its input includes `type: z.enum(ENTITY_TYPES)` plus a flat
superset of common and cron-only fields. The caller must choose the stored
discriminator before expressing intent.

`packages/server/src/tools/backlog-update.ts:9-35` repeats the pattern for
updates. One schema carries task status/evidence, parent relations, milestone
dates, artifact content type, and cron scheduler state. The description even
redirects body edits to another generic write surface.

Both tools are registered unconditionally in
`packages/server/src/tools/index.ts:58-65`.

### The core has one operation wrapped around a second factory

`packages/server/src/core/create.ts:30-67` exports `createItem`. It allocates
an ID, invokes the lower factory, persists the entity, captures eligible
episodic memory, and records the mutation. This is the actual create
operation, but its name retains the vocabulary ADR 0106.1 retired.

`packages/server/src/storage/entity-factory.ts:68-78` separately exports
`createEntity`. It stamps time, strips `undefined`, and parses the shared
`EntitySchema`. It is a synchronous construction helper, not the complete
business operation.

There is no live `createTask` or `CreateTaskInput` implementation. Those
fossils were removed by ADR 0106.2. Phase 2 therefore needs a precise
convergence, not another rename sweep: the complete core operation becomes
`createEntity`, while the lower helper receives a construction name and is no
longer confused with the write funnel.

### Update is transport-free but accepts the leaked superset

`packages/server/src/core/update.ts:22-71` correctly owns persistence,
whole-entity schema validation, memory capture, and journaling. Its
`UpdateParams`, however, mirrors the generic MCP superset
(`packages/server/src/core/types.ts:152-170`). The new intent executor should
use this core primitive for the first safe algebra rather than duplicating its
write mechanics.

The low-level CLI already calls core directly:

- `packages/server/src/cli/commands/create.ts:6-28`;
- `packages/server/src/cli/commands/update.ts:5-29`.

That makes the CLI the smallest honest escape hatch for maintainers and scripts
without preserving a second agent-facing contract.

### A dead platform constraint leaks into a vision-path tool

`packages/server/src/tools/backlog-create.ts:31-40` branches on an optional
`resolveSourcePath` dependency and returns “not supported in cloud mode.” D1 is
a constrained satellite under ADR 0104 and is not a design target for this
surface. New intent tools must not carry cloud guards. Local file reading stays
in explicitly local adapters such as the CLI; the core receives content.

### The journal assumes the legacy tool names

The write journal currently couples semantic attribution to four fixed names:

- `packages/server/src/operations/types.ts:12-32` closes `ToolName` and
  `WRITE_TOOLS` over `backlog_create`, `backlog_update`, `backlog_delete`, and
  `write_resource`;
- `packages/server/src/core/operation-log.ts:21-59` derives the live event type
  from that name;
- `packages/server/src/operations/resource-id.ts:8-43` selects ID extraction by
  exact legacy tool name and has a hard-coded ID-prefix regex.

Deleting the old tools without changing this seam would either lose the
semantic verb in activity history or require adding every future intent to
closed switch tables. The journal needs a stable mutation classification
separate from the human-visible tool name.

### ADR 0113 has designed the source of truth but it is not live yet

The packaged definitions in `packages/shared/src/substrates/*.ts` currently
contain schema, structure, hints, and UI metadata, but no intent declarations.
ADR 0113 R7 requires explicit declarations compiled through a closed safe
algebra; Phase D assigns the server executor and MCP registration seam.

Basalt, owner of ADR 0113, and this ADR agree on the ownership boundary:

- shared/0113 owns declarations, compiler validation, and compiled DTOs;
- server/0106.5 owns the consumer port, core executor, and MCP registration;
- neither side creates a parallel registry contract.

ADR 0113 clarification commit `d2d3893` is the authoritative producer ruling
for the bounded `set-field` operation and the compensated,
explicitly-non-atomic `relate-and-transition` plan consumed below.

## Rulings

### R1. Shared compiles intent declarations; the server consumes opaque DTOs

The source definition is
`RuntimeSubstrateIntentDefinition` in
`packages/shared/src/substrates/runtime-substrate.types.ts`. It is a
discriminated union with common declaration data:

```ts
{
  verb: string;
  toolName?: string;
  description: string;
  requiredInputs: string[];
  optionalInputs?: string[];
  defaults?: JsonData;
}
```

Its mutation operation is one of:

- `create`;
- `transition { transition }`;
- `set-field { field, value }`;
- `relate { relation }`;
- `append-relation { relation }`;
- `relate-and-transition { relation, transition, targetTransition? }`.

`set-field` is intentionally narrower than update. V1 accepts one canonical
scalar property and one literal scalar value. The compiler rejects `id`,
`type`, the substrate's workflow field, and declared relation fields; the
executor still validates the whole resulting entity through the active
registry. This exists for state such as Cron `enabled` that is explicitly not a
workflow status.

The shared compiler validates declared fields/defaults, referenced
transitions/relations, tool-name syntax, and global collisions. It emits an
opaque safe DTO with a required `toolName`. Its default derivation is
`backlog_${verb}_${substrateType}`; a declaration may provide an explicit
override when the semantic verb is already globally specific. The server never
derives a name from `verb` or substrate type.

The server declares an `IntentRegistryPort` over those compiled DTOs. It does
not import project definitions, interpret source configuration, or recreate
compiler checks.

The frozen ADR 0113 producer seam sharpens that boundary:

- the compiled DTO carries `intentInputSchema`, explicit input-to-field
  bindings, create-only `fixedFields`, resolved transition fields/states, and
  relation cardinality plus allowed target substrate types;
- two-entity operations name `sourceInput` and `targetInput` explicitly;
  declaration input order is never semantic;
- defaults exposed as optional inputs compile into the input schema, while
  non-input create defaults compile into `fixedFields`; there is no second
  defaults bag for the server to interpret;
- the registrar receives only the list-only `IntentRegistryPort`, while the
  executor receives a separate one-method `IntentWriteValidatorPort`;
- the compiler receives the server's complete reserved MCP name set so project
  declarations cannot shadow read, memory, write, or retired tool names.

Search and recall descriptors stay deferred until ADR 0113 Phase C proves a
need. They are not smuggled into a mutation executor.

|---|---|---|---|
| Task | `backlog_create_work` | create | title; optional content, parent, references |
| Task | `backlog_start_task` | transition | id |

| Epic | `backlog_plan_epic` | create | title; optional content, parent, references |
| Folder | `backlog_organize_folder` | create | title; optional content, parent |
| Artifact | `backlog_attach_artifact` | create | title, parent; optional content, path, content type, references |
| Milestone | `backlog_target_milestone` | create | title; optional content, parent, due date, references |
| Cron | `backlog_schedule_cron` | create | title, schedule, command; optional content, parent, enabled |
| Cron | `backlog_pause_cron` | set-field (`enabled=false`) | id |

definitions; this table states semantic scope, not a second schema authority.

`backlog_create_work`, `backlog_capture_requirement`, and
`backlog_capture_prompt` are explicit declaration overrides because their
semantic verbs are already globally specific. The other names use the shared
compiler's default naming rule. Memory remains
`backlog_remember`/`backlog_forget`.

No other verbs ship in this phase. In particular, there is no generic
`backlog_transition`, `backlog_relate`, or auto-generated status matrix.

### R3. The generic tail escape hatch is local and explicit

`backlog_create` and `backlog_update` are deleted from the MCP surface. They are
not renamed, hidden, deprecated, or regenerated.

The CLI `backlog create` and `backlog update` commands remain as the
maintainer/script escape hatch and continue to call transport-free core. They
gain one deliberately low-level `--fields <json-object>` option so the tail can
carry substrate-specific fields (including `null` clears on update) without
growing one CLI flag per substrate. Existing common flags stay for ergonomics.
A docs-native substrate can also be written as a human-visible file and
reconciled by the runtime.

This is deliberately asymmetric:

- agents get the safe, discoverable intent port;
- operators retain a low-level local escape hatch;
- the server does not carry two competing MCP dialects.

Granite's delegated founder ruling on 2026-07-16 confirms that this CLI-only
escape hatch satisfies founder intent for this phase: the CLI is a first-class
consumer under “one core, many consumers,” while docs-native files and CLI
cover the undeclared/rare tail without dynamic MCP machinery.

If real use later demonstrates that agents need a tail mutation absent from the
declared verbs, the smallest response is to declare that repeated intent. A
new generic MCP write port requires a new ADR and evidence; it is not reserved
speculatively here.

### R4. Core owns one complete `createEntity` operation

`packages/server/src/core/create.ts` exports the sole complete
`createEntity(service, normalizedParams, ctx, attribution)` operation. The
intent executor resolves the compiled intent, substrate type, declared
defaults, and input mapping before calling this reusable funnel. The CLI passes
an explicit substrate type and fields directly. The operation owns:

1. ID allocation;
2. entity construction and schema validation;
3. persistence;
4. episodic capture;
5. mutation journaling and live notification.

The synchronous helper now named `storage/createEntity` is renamed to a
construction term such as `buildEntity`. It may remain a narrow validation
helper, but it is not exported or documented as a second create operation.

`CreateParams` becomes `CreateEntityParams`. It carries an explicit substrate
type plus normalized fields broad enough for the CLI and intent executor, but
no MCP tool exposes that type directly.

No per-substrate create function is added.

### R5. One core intent executor implements the closed algebra

`executeSubstrateIntent` is transport-free. It accepts a compiled intent,
validated input, the active backlog service, and a `WriteContext`.

It dispatches only the mutation algebra from R1:

- create maps declared/defaulted fields into `createEntity`;
- transition maps a compiled transition to `updateEntity`;
- set-field applies the compiler-owned fixed scalar assignment, then validates
  the whole postimage through the active registry;
- relation operations validate both endpoints and update the declared relation;
- combined operations follow the explicit multi-entity semantics in R12; they
  do not rely on nonexistent service transactions.

The executor contains operation mechanics, not substrate-specific name
switches. Meaning such as `accept → status=accepted` or
`supersedes → relation field` arrives in the compiled DTO.

Project declarations cannot run JavaScript, provide callbacks, select files, or
issue arbitrary patches.

Only algebra cases exercised by the initial sixteen verbs are implemented in
this phase. The shared declaration union may name future-safe cases required by
ADR 0113, but `relate` and `append-relation` do not receive speculative server
execution paths until a shipped verb uses them.

The registrar makes that deferral visible rather than exposing a tool that can
only fail. An intent using either unsupported operation is quarantined with the
reason `operation kind not yet executable — 0106.5 R5 initial-16 scope` and an
escape path stating that the first real project declaration needing
`relate`/`append-relation` triggers implementation. It is never silently
skipped. This consumer-side capability gate leaves the frozen shared compiler
contract unchanged.

### R6. Generated MCP adapters are schema mappers, nothing more

`registerSubstrateIntents` iterates the port's compiled registry and registers
each declared tool using its required `toolName`, description, and compiled
input schema/projection.

Each handler:

1. accepts already-described inputs;
2. builds `WriteContext`;
3. calls `executeSubstrateIntent`;
4. formats a result or domain error.

It does not derive names, inspect Zod substrate schemas, branch on type, read
files, choose transitions, or implement business rules.

### R7. Legacy create/update disappear in the same engineering slice

When all sixteen verbs can register from the compiled registry:

- delete `packages/server/src/tools/backlog-create.ts`;
- delete `packages/server/src/tools/backlog-update.ts`;
- remove both imports and registrations from
  `packages/server/src/tools/index.ts`;
- delete tool-specific tests and replace them with registry, executor, and
  generated-adapter tests;
- update README/AGENTS/tool documentation in the same change.

There is no feature flag or compatibility period. No failure silently restores
the generic tools.

### R8. Vision-path tools contain no cloud-mode guards

No generated intent handler branches on Worker/D1 capability. In particular,
the `source_path` behavior and “not supported in cloud mode” response in the
old create tool are deleted with that tool.

File-to-content resolution remains a local adapter concern. The core and
intent executor operate on supplied data. D1 receives no parity work in this
phase.

### R9. Journal identity and mutation classification are separate

Every core write receives explicit attribution:

```ts
interface MutationAttribution {
  tool: string;
  mutation: 'create' | 'update' | 'delete' | 'resource-edit';
}
```

The operation log records:

- `tool`: the semantic public name that initiated the action;
- `mutation`: a stable class such as `create`, `update`, `delete`, or
  `resource-edit`;
- `resourceId`: supplied by the core result or validated input, not recovered
  through a tool-name-specific regex.

Live event mapping uses `mutation`, not a closed list of tool names. The viewer
renders the semantic `tool` and groups/styles by `mutation`.

For single-entity operations, the top-level core write remains the journal
owner. The intent executor passes semantic attribution into
`createEntity`/`updateEntity` and does not append a wrapper entry. The CLI
passes its own stable command name.

For a compiled multi-entity plan, the executor is the one journal owner. It
uses a narrow internal validated-save primitive that does not journal
intermediate writes, then records/emits the semantic success only after the
complete plan succeeds. This exception prevents a failed compensated
`backlog_supersede_adr` from appearing as a completed semantic action. New
writes never emit `backlog_update`.

Existing JSONL entries without `mutation` remain readable through a small
legacy inference at the read boundary. New writes never emit the old generic
tool names.

This keeps activity history truthful without teaching journal infrastructure
every future substrate verb.

The local JSONL journal is the vision-path contract. The descoped D1 satellite
retains its existing table and legacy-name inference; semantic intent tools are
not registered there, so this phase adds no D1 column, migration, or parity
work. If remote mode returns under a local-filesystem/VPS design, its journal
must persist `mutation` before receiving semantic tools.

### R10. Intent registration is required, with per-definition quarantine

The local runtime supplies a compiled intent registry to `registerTools`.
Read-only tools may register without it only in explicitly read-only
compositions. A normal writable runtime with no readable base compiled registry
fails startup visibly; it does not expose a partial write surface.

ADR 0113 R9 still governs individual project definitions: an invalid definition
is quarantined with a diagnostic while valid definitions and packaged
substrates continue. Its invalid intents are absent; they do not poison the
entire writable server. Name collisions among otherwise valid active intents
are compiler diagnostics and the colliding definitions are quarantined.

Read surfaces (`backlog_list`, `backlog_get`, `backlog_search`,
`backlog_recall`) are outside this replacement.

### R11. Measure tool shape, do not build a tool framework

The implementation records the generated tool count and serialized input
schema size in a deterministic unit test. The purpose is regression visibility,
not a runtime budgeter or configuration system.

Adding a new algebra case, generic fallback, or manifest-tuning framework is
out of scope until a real substrate cannot express a real intent.

### R12. Multi-entity intents state their consistency limit honestly

The current service has atomic single-file writes but no cross-entity
transaction or mutation mutex. `relate-and-transition` therefore cannot claim
database-style atomicity.

The initial `backlog_supersede_adr` implementation must:

1. preload both entities and check the declared relation/transition
   preconditions;
2. construct and registry-validate both postimages before the first save;
3. write the replacement/source `supersedes` relation first;
4. transition the older/target ADR to `superseded` second;
5. if the second save fails, attempt to restore the first entity from its
   captured pre-image;
6. report whether compensation succeeded; compensation failure returns an
   explicit `partial_failure` carrying both IDs;
7. journal and emit the semantic success only after both writes succeed.

This is explicit compensating consistency, not a transaction. A general batch
write API or lock is out of scope. If real concurrent use demonstrates that
compensation is insufficient, that pressure justifies a storage-level atomic
mutation design; this ADR does not pre-build one.

Retry is idempotent: relation append deduplicates, and an already-completed
matching target transition succeeds.

## Rejected options

### Keep generic create/update as aliases

Rejected. It preserves the exact storage dialect this ADR removes, creates two
ways to perform the same write, and contradicts ADR 0114's no-legacy migration
discipline. Compatibility is not a product requirement for our own unversioned
tool surface.

### Generate CRUD from every substrate schema

Rejected. A schema cannot name intent. This recreates the founder's feared tool
bloat and turns adding a substrate into an uncontrolled public API expansion.

### Add one `backlog_execute_intent` tool with `verb` and `type`

Rejected. It merely moves the discriminator leak into a new envelope and hides
each intent's exact input shape from Tool Search.

### Keep a new registry-backed generic MCP fallback

Rejected for this phase. It would be safer than the legacy flat schemas but
would still create a parallel low-level agent contract without evidence. The
CLI and docs-native files already cover the tail. Escalation trigger: live
usage repeatedly shows agents need MCP writes for undeclared substrates. Only
then is this option designed against the Phase D runtime registry that actually
exists at that time.

### Hand-write one MCP module per verb

Rejected. It duplicates mechanical registration and invites business logic
into transport files. Explicit declarations plus one generated adapter are the
smallest way to support built-in and user-defined substrate intents.

## File-level engineering plan

### Phase A — converge declaration-agnostic core seams

1. Rename `core/create.ts:createItem` to the complete `createEntity` operation
   and `CreateParams` to `CreateEntityParams`; rename the touched
   `core/update.ts:updateItem`/`UpdateParams` fossil to
   `updateEntity`/`UpdateEntityParams`; update CLI and tests.
2. Rename the storage construction helper to `buildEntity` and narrow its
   exports.
3. Change operation logging to accept semantic tool name + mutation class +
   explicit resource identity.
4. Update operation readers and viewer activity formatting with legacy-read
   inference only.
5. Add the CLI's low-level `--fields <json-object>` tail input.

This phase can land after Beryl approval without inventing any ADR 0113
declaration type.

### Prerequisite — land the ADR 0113 open-write and compiler seams

The runtime-substrate verbs require ADR 0113 Phase B's open entity type,
registry-backed validation, identity allocation, and docs-native storage
contract. They cannot run on the current closed `Entity`/`EntityType` service.

After that foundation, the ADR 0113 owner lands the first Phase D slice:

1. Add `runtime-substrate.types.ts` and the closed mutation union under
   `packages/shared/src/substrates/`.
2. Add intent declarations to packaged substrates, including exact tool names
   from R2.
3. Compile declarations into opaque safe DTOs; validate fields, defaults,
   transitions, relations, name syntax, and collisions.
4. Export only the compiled contract needed by the server port.

0106.5 consumes this phase; it does not duplicate it.

### Phase B — generate the intent port and cut over in one slice

1. Add the consumer-owned `IntentRegistryPort`.
2. Add `core/substrates/execute-substrate-intent.ts` plus satellite types,
   implementing only algebra cases exercised by R2.
3. Add `tools/register-substrate-intents.ts`.
4. Wire the compiled registry into local server composition.
5. Delete `backlog-create.ts` and `backlog-update.ts` and remove registration.
6. Remove the obsolete `resolveSourcePath` dependency from `ToolDeps` if no
   remaining MCP consumer needs it.
7. Keep CLI create/update as low-level local adapters.
8. Update README, AGENTS.md, tool snapshots, and examples.

### Phase C — validate the real surface

Unit tests use the repository's global memfs setup and cover:

- compiler rejection of collisions, invalid fields/defaults, and unknown
  transitions/relations;
- exact registration of the sixteen initial verbs and absence of
  `backlog_create`/`backlog_update`;
- every algebra branch exercised by the initial surface through the real core
  executor;
- one complete create operation, including validation, persistence, memory
  capture, and journaling;
- transition, relation, and compensation failure behavior;
- semantic tool names plus stable mutation classes in operation history;
- legacy JSONL read compatibility;
- writable startup failure for a missing/corrupt base registry and quarantine
  for one invalid project definition;
- generated manifest count/shape snapshot.

Manual validation, required by the Development Loop:

1. start the local MCP server in a fresh project;
2. confirm Tool Search discovers intent verbs from their descriptions;
3. create, start, block, and complete work through the semantic tools;
4. create one packaged docs-native substrate entity;
5. inspect the markdown entity and activity journal;
6. confirm generic MCP create/update are absent;
7. exercise the CLI escape hatch once;
8. restart and confirm the same compiled surface returns deterministically.

## Sequencing and dependency

Design can land before ADR 0113 Phase D, but the MCP cutover cannot land against
a hand-copied DTO. Basalt owns the declaration/compiler implementation.

The approved sequencing is:

1. this branch independently lands only the declaration-agnostic core
   `createEntity` naming and journal mutation-class refactor;
2. ADR 0113 lands the authoritative declaration and compiled-intent DTO as its
   first Phase D slice, after Phase C (or as an extracted prerequisite only if
   Granite explicitly accelerates 0106.5);
3. this branch rebases its registration/deletion slice onto that commit and
   consumes compiled `toolName` opaquely.

The dependency is intentional and narrow. No intent DTO, generated
registration, or legacy MCP deletion lands before the authoritative compiler
contract. Once it does, the full R2 set and same-change cutover land together.

## Consequences

### Positive

- Agents select a domain action rather than a storage discriminator.
- Tool schemas disclose only the shape needed for the selected intent.
- Built-in and user-defined substrates share one explicit declaration path.
- Core write behavior remains reusable by MCP, CLI, and future local consumers.
- Operation history becomes extensible without losing the initiating verb.
- D1 constraints stop shaping a local-first vision surface.

### Cost

- Sixteen public names require deliberate documentation and stable discovery
  descriptions.
- The shared compiler becomes a startup dependency for writable runtimes.
- Activity-log readers need a small compatibility path for old entries.

### Risk controls

- Verb sprawl: declarations are explicit; no inference; additions require a
  real use case.
- Adapter drift: one generated registrar and one executor.
- Partial cutover: generic MCP tools are deleted only when the entire initial
  compiled set is present.
- Framework growth: the server implements only algebra cases exercised by the
  initial verbs.

## Exit criteria

Phase 2 is complete when:

1. the sixteen initial tools are compiled from substrate declarations;
2. `backlog_create` and `backlog_update` do not exist in the MCP registry or
   live tool source;
3. all create intents use core `createEntity`;
4. all other intents use the one safe executor;
5. the operation journal preserves semantic tool name and stable mutation
   class;
6. no vision-path tool contains a D1/cloud capability guard;
7. CLI generic create/update remain functional as the explicit local tail;
8. unit tests, typecheck, build, and the manual local loop are green.
