# 0092.12. Demand-Aware Consolidation Ripeness

**Status**: Proposed
**The thread closes a loop on itself**: ADR 0092.7 deferred recall-demand
ripeness because no demand data existed; ADR 0092.9 R-16 created it (every
recall appends `{ts, type:"recall", query, ids}` to `memory-usage.jsonl`);
this ADR feeds it back into bundle scoring. Evidence basis (0092.5 §4b /
Hindsight docs): *"frequently-asked recall themes are the strongest
mental-model candidates"* — demand is the strongest ripeness signal, ahead
of count and age.

## Design

- **Pure demand fold**: `demandCounts(jsonlLines, {windowDays: 30, now})` →
  Map<MEMO-id, recall-event count>. Deterministic, replayable, no state.
- **Bundle demand** = number of recall events (within the window) that
  returned **at least one member** — event-level, not id-level, so one
  recall touching 3 members counts once (Fine-Mem's participation dilution
  in spirit).
- **Ripeness becomes**: `count ≥ min_count AND (oldest_age ≥ min_age_days
  OR demand ≥ min_demand)` (default min_demand: 3). Age OR demand: a young
  bundle the agent keeps asking about is ripe — being asked about *is* the
  maturity signal. Count still gates (one hot memory is not a cluster).
- **IO at the edge** (ADR 0090 pattern, same as wakeup's `readIdentity`):
  core takes an injected `readUsageLines?: () => string[]`; the Node
  bootstrap reads the JSONL; absent reader → demand 0 everywhere, behavior
  identical to Phase D (graceful degradation, Worker-safe).

Bundles gain a `demand` field; ordering becomes ripe → demand → count.
The CLI renders demand so the human sees *why* a bundle is ripe.
