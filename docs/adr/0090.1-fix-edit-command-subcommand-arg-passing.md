# 0090.1. Fix edit Command — Move `<id>` to Subcommands

Continuation of [ADR-0090](./0090-cli-tool-and-core-extraction.md) § "edit Command — subcommand-per-operation pattern".

## Problem Statement

The `edit` CLI command is broken. Commander.js cannot resolve `<id>` as a positional argument on a parent command that also has subcommands.

- ADR-0090 designed: `edit <id> replace <old> <new>`
- Commander.js parses `edit TASK-0641 append "text"` by treating `TASK-0641` as a subcommand name lookup, not as the `<id>` argument
- `edit.args[0]` inside subcommand actions is always `undefined` because Commander consumed the positional token during subcommand dispatch

### Root Cause

Commander.js subcommand dispatch works by matching the first token after the parent command name against registered subcommand names. When the parent has `<id>` as a required arg AND subcommands, Commander sees `TASK-0641` and tries to find a subcommand named `TASK-0641` — it doesn't parse it as the `<id>` arg first.

This is a known Commander.js limitation: parent positional args and subcommands don't compose. The parent's `.argument()` is only parsed when no subcommand matches.

### Evidence

Observed in the transcript: multiple attempts to call `npx backlog-mcp edit TASK-0641 append "text"` all fail silently or produce unexpected behavior. The `edit --help` output shows `edit <id>` but the subcommands never receive the id.

## Proposed Solutions

### Option A: Move `<id>` to each subcommand (verb-first)

```
edit append <id> <text>
edit replace <id> <old> <new>
edit insert <id> <line> <text>
```

- Subcommands own all their arguments — standard Commander.js pattern
- Verb-first reads naturally: "edit append TASK-0641 text"
- Each subcommand's `--help` is self-documenting
- **Breaking change**: syntax order changes from `edit <id> <verb>` to `edit <verb> <id>`

### Option B: Drop parent `<id>`, use `--id` option on parent

```
edit --id TASK-0641 append "text"
edit --id TASK-0641 replace "old" "new"
```

- Options are parsed before subcommand dispatch, so `--id` works
- Accessible in subcommands via `edit.opts().id`
- Inconsistent with every other command (`get <id>`, `update <id>`, `delete <id>` all use positional)

### Option C: Abandon subcommands, use `--op` flag

```
edit TASK-0641 --append "text"
edit TASK-0641 --replace "old" "new"
```

- Keeps `<id>` positional on the parent
- Mutually exclusive flags are messy — Commander doesn't enforce exclusivity natively
- Different arg counts per operation make flag validation complex
- ADR-0090 explicitly rejected this: "if two flags would be mutually exclusive and have different argument shapes, they should be subcommands"

## Decision

**Selected: Option A** — move `<id>` as first argument of each subcommand.

```
edit replace <id> <old> <new>
edit append <id> <text>
edit insert <id> <line> <text>
```

### Rationale

- Standard Commander.js pattern — subcommands own their arguments, no hacks
- Verb-first is natural: matches `git remote add <name> <url>`, `docker volume inspect <name>`
- Each subcommand is self-contained with correct `--help`
- ADR-0090's design intent was subcommand-per-operation — this preserves that, just fixes arg placement
- The parent `edit` command becomes a pure namespace with no arguments

### Assumptions that must hold

- No existing automation depends on `edit <id> <verb>` syntax (the command was broken from the start, so nothing could depend on it)
- Verb-first ordering is intuitive enough for both agents and humans

### Trade-offs

- Syntax changes from the (broken) `edit <id> append` to `edit append <id>` — acceptable since the original never worked

## Consequences

- **Positive**: `edit` subcommands work correctly; each subcommand is self-documenting
- **Negative**: None — the previous syntax never worked, so no backward compatibility concern
- **Risk**: None identified

## Implementation Notes

- Remove `<id>` from parent `edit` command definition
- Add `<id>` as first positional arg on each subcommand (`replace`, `append`, `insert`)
- Subcommand actions receive `id` directly from Commander arg parsing — no `edit.args[0]` hack
- Parent `edit` command becomes: `.command('edit').description('...')`
