# 0106.1. Domain Vocabulary — Substrate, Entity, and Projection

**Date**: 2026-06-17
**Status**: Accepted
**Backlog Item**: TASK-0688
**Thread**: substrate architecture (0098) · supports the port/core vision (0106)

## Why this ADR exists

The codebase uses **four words for one concept** and lacks a settled rule for
the two foundational nouns. This is the root of the "incohesive naming" friction.
A glossary outlives any single design, so it gets its own ADR that other ADRs and
reviews can point at. This is a **definitions decision**, ratified by the
maintainer; the mechanical renames that follow are tracked separately.

## The two foundational nouns (ratified)

> **A Substrate is the low-level definition of a type. An Entity is an instance
> of one specific Substrate.**

This is the **class/instance** (schema/row) relationship. It is not a preference —
it is what the code already encodes; this ADR only names it as a rule.

### Substrate — the definition (the abstraction)

- A `SubstrateDefinition` (`shared/src/substrates/base.ts`) *describes* a type:
  `type`, `prefix`, `label`, `schema`, `structure`, `extraFields`, `hint`, `ui`.
- Its own doc comment: *"One declaration per entity type. Drives TS types,
  validation, storage invariants, viewer/agent metadata."*
- **Exactly 7, singletons, design-time**: `TaskSubstrate`, `EpicSubstrate`,
  `FolderSubstrate`, `ArtifactSubstrate`, `MilestoneSubstrate`, `CronSubstrate`,
  `MemorySubstrate`. You never have "two Task substrates"; you never
  `createSubstrate`.
- Lives in `@backlog-mcp/shared`. It is the **low-level abstraction**.

### Entity — the instance

- `Entity = z.infer<typeof EntitySchema>` — a parsed object with a real `id`,
  `title`, and data. `TASK-0042` is an Entity; `MEMO-0011` is an Entity.
- **N of them** (every stored row) — the runtime currency (519 usages of
  `Entity*` across the source, vs. 7 substrates).
- A type-narrowed Entity uses the substrate's name as a noun: `Task`, `Memory`,
  `Cron` (the `z.infer` aliases). These mean *"an Entity of that substrate"* —
  never the definition.

### The bridge

- `getSubstrate(type: EntityType): SubstrateDefinition`
  (`shared/src/substrates/registry.ts:70`) maps an **instance's** discriminator to
  its **definition**. The function's existence is the proof the model is
  class/instance; this ADR just makes it the rule.
- `EntityType` (`'task'`, `'memory'`, …) is the **discriminator** that links an
  Entity to its Substrate.

## The third tier — Projection (DTO)

Not every value-object is an Entity. Response shapes are **projections** of an
Entity for a specific port call, and must NOT be renamed to `Entity`:

- `RecallItem` carries `digest`, `score`; `SearchResultItem` carries match
  metadata; `ListItem`/`GetItem` are list/detail views. None are storable
  entities — they are shaped views.
- Rule: **`*Result` / `*Item` = a projection**, never a synonym for Entity.

## Canonical glossary

| Concept | Canonical word | Cardinality | Home | Lifetime |
|---|---|---|---|---|
| Definition of a type (the abstraction) | **Substrate** | 7 (one per type) | `shared/substrates/` | design-time |
| Type discriminator value | **EntityType** | 7 enum members | `shared/entity-type.ts` | the link |
| A stored instance | **Entity** | N | `shared` (runtime currency) | runtime |
| A type-narrowed instance | substrate noun: **Task / Memory / Cron** | — | `z.infer` aliases | runtime |
| A shaped response view | **Projection** (`*Result` / `*Item`) | — | port contracts | per-call |
| Request payload to a port | **`*Request`** (or `*Params`) | — | port contracts | per-call |

## Banned synonyms (one concept, one word)

- **"Item" as a synonym for Entity** — retired. `createItem` → `createEntity`.
  "Item" is permitted ONLY for projections (`ListItem`, `RecallItem`).
- **"Task" as a synonym for Entity** — retired. `Task` means the task substrate
  *only*. `CreateTaskInput` → `CreateEntityRequest`. `TaskStorage` /
  `taskStorage` / `task: Entity` params (`storage/backlog-service.ts`) are
  fossils from when tasks were the only type — they store/return `Entity`, not
  `Task`.
- **"Substrate" as an instance** — never. No `createSubstrate`; substrates are
  read via `getSubstrate`.

## Evidence of the current drift (what the rename will fix)

- `storage/schema.ts`: `createTask` / `CreateTaskInput` build and return the
  discriminated `Entity` (folders, crons, milestones) — task-named, entity-typed.
- `storage/backlog-service.ts`: imports `Entity`, but uses `TaskStorage`,
  `taskStorage`, and `task: Entity` / `allTasks` throughout — `Task` used as
  "Entity".
- `core/types.ts`: `CreateParams`/`CreateResult` but the created thing is an
  `Entity`; `ListItem`/`GetItem` are projections (correctly *not* entities).

## Decision

1. **Substrate** = definition (7, design-time, `shared`). **Entity** = instance
   (N, runtime). This class/instance rule is canonical.
2. **Projection** is a distinct third tier (`*Result`/`*Item`); never collapse it
   into Entity.
3. Retire "Item"/"Task" as synonyms for Entity (see banned list). One concept,
   one word.
4. This glossary is the source of truth; ADR 0106 and code reviews reference it.

## Consequences

**Positive**
- A single vocabulary across shared/core/storage/port — the incohesion source is
  removed.
- New contributors learn the model from one bridge fact (`getSubstrate`) + this
  table.
- Enables the 0106 port work: intent verbs map to `createEntity` (not
  `createItem`/`createTask`), and DTOs are honestly named projections.

**Negative / cost**
- A wide mechanical rename (`createTask`→`createEntity`,
  `CreateTaskInput`→`CreateEntityRequest`, `TaskStorage`→`EntityStorage`,
  `task`→`entity` locals) touching storage + core + ~50 test sites.
- Risk of over-rename: do NOT rename genuine `Task`-substrate references or
  projection `*Item` types. The rename must be glossary-driven, not blind
  search/replace.

## Execution (separate from this decision)

The glossary is settled now. The mechanical renames are a follow-up refactor,
done glossary-driven (audit each `Task`/`Item` site: is it the task substrate, a
generic entity, or a projection?), with typecheck + full server suite as the gate.
Tracked under TASK-0688 alongside the 0106 port work.
