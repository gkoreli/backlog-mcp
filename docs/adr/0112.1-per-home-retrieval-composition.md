# 0112.1. Per-Home Retrieval Composition — Indexes, Cross-Home Fusion, and the Store-Boundary Seam

**Status:** Proposed — cross-thread seam ruling, authored by the search &
memory domain seat ahead of ADR 0112 Phase D so Phase D is built against a
fixed contract instead of discovering one mid-implementation.

## Context

Three in-flight threads meet at one seam:

- **ADR 0112** rules that a backlog has multiple homes, each with the same
  shape (R-1), that hidden state is only control or derivation (R-3), and
  that cross-home reads are explicit, read-only, and rank-fused (R-9).
  Phase D — per-home indexes, per-home memory, the usage overlay, and
  `home: all` — is unstarted.
- **ADR 0115** ratified the store-boundary law (its R-5): provenance and
  usage signals reach read surfaces only via
  `BacklogMemoryStore.toMemoryEntry` → `MemoryEntry`. Its known exception
  (`core/wakeup.ts` reading raw entities) is being closed as a sequenced
  follow-up.
- **ADR 0116** rules that fusion changes are fixture-gated (R-1, R-3), that
  cross-home merging is RRF over within-home ranks with deterministic ties,
  and that readiness is per-home, never process-global (R-4).

This ADR fixes the composition contract those threads share. It rules the
retrieval side only; ADR 0112 keeps home discovery, identity, layout, and
the overlay file format.

## Rulings

### R-1 — The composition unit is the home runtime, instantiated lazily

Each home owns one vertical: storage → search service (index, cache,
readiness state) → memory store → usage state. Nothing retrieval-shaped is
process-global except stateless compute (R-2). Home runtimes are keyed by
`home_id` and instantiated on first touch — booting a server in a project
must not build indexes for every home the machine has ever seen.
`home: all` instantiates only the homes the read selects.

This extends ADR 0112 R-1 to the retrieval graph and retires the module
singletons that currently pin one index to one process
(`packages/server/src/storage/local/backlog-service.ts:26-30`,
`packages/server/src/memory/bootstrap.ts` module-level paths).

Per-home caches keep the existing envelope rules: `INDEX_VERSION` and (once
ADR 0116 R-7 lands) the embedding-model fingerprint are validated per home,
and a mismatch rebuilds that home only.

### R-2 — The embedding model is process-wide; embedding work is globally bounded

One Transformers.js pipeline serves all homes: the model is stateless
compute, and duplicating ~90 MB of weights per home would be absurd. What
is per-home is the *output* (vectors inside that home's index) and the
*readiness state* (ADR 0116 R-4).

Background semantic preparation, if Phase 1B is earned, draws from a single
process-wide bounded work queue with per-home fairness — N homes must not
multiply into N concurrent embedding storms. One home's model download or
failure state is shared (there is one model), but one home's *index build*
never blocks another's queries.

### R-3 — `home: all` merges what each home can serve now

The merger consumes only within-home final ranks. Each home completes its
own full pipeline first — for search: retrieval, fusion, decay,
coordination, pin; for recall: additionally the post-filters and
`usageFactor`. Cross-home merge is RRF over those ranks with the
deterministic tie order ruled in ADR 0116 R-3 (RRF score, then `home_id`,
then stable local ID; complete tiers at the limit, bounded by one extra
result per home).

Availability rule: the merge never waits for a home's semantic readiness.
A home in `lexical_ready` serves its BM25-only ranking; a home that cannot
serve at all (still initializing, degraded) is reported in the response's
per-home status with a reason — absent silently is forbidden. Every result
carries `home`, `home_id`, `source_path`, and its within-home rank; raw
scores from different homes are never compared or presented as comparable.

### R-4 — Memory verbs across homes

- **recall** defaults to the resolved home; `home: all` is allowed
  (read-only) and merges per R-3. `usageFactor` is within-home ranking and
  is applied before the merge. Expand/cite usage events attribute to the
  home that owns the memory.
- **wakeup** is home-scoped orientation. Under `home: all` it groups by
  home (ADR 0112 R-9); the knowledge section stays a per-home top-5 — there
  is no cross-home knowledge fusion. Orientation is about *this* project's
  authority, not a blended one.
- **remember / forget** are single-home writes; `home: all` writes are
  rejected (ADR 0112 R-9, reaffirmed).
- **consolidation** is per-home only: bundles never mix homes, demand
  counts read that home's `state/memory-usage.jsonl`, and the consolidator
  watermark (`state_key: consolidation.watermark.<scope>`) is a per-home
  fact.
- **contradictions**: `state_key` uniqueness is a per-home invariant. The
  same `state_key` live in two homes is not a contradiction — two projects
  legitimately hold different values for the same key. Cross-home
  contradiction detection is deferred until a real case demands it.

### R-5 — The overlay swap is contained at the store boundary

The Phase D contract for usage state, restating ADR 0115 R-5 as
implementation constraints:

- **Project homes**: `usage_count` / `last_used_at` live in the home's
  `.backlog-mcp/state/` usage overlay (ADR 0112 R-3 — reads must not dirty
  the repository). The overlay is read and merged *only* inside
  `BacklogMemoryStore.toMemoryEntry`. `MemoryEntry`'s shape, `RecallItem`'s
  stub fields, and `usageFactor` semantics do not change. The overlay file
  format is ADR 0112's ruling to make.
- **Global home** (`~/.backlog`): keeps the shipped frontmatter behavior
  (ADR 0092.9). It is not a repository; dirtying-reads do not apply, and
  zero-migration wins.
- **Precondition**: the `core/wakeup.ts` raw-entity read must be closed
  (sequenced as onyx's follow-up to ADR 0115) before Phase D ships —
  otherwise project-home wakeup reports `uses: 0` from frontmatter that no
  longer holds usage truth.

### R-6 — Cross-home citations are not counted until identity says they can be

`recordCitations` scans written memory bodies for `MEMO-` references and
bumps usage on the cited memory. Unqualified cross-home IDs are an error by
ADR 0112 R-9, and the qualified-identity grammar is ADR 0112 R-6's seam.
Until qualified cross-home references exist, citation events resolve within
the writing memory's home only; a reference that does not resolve there is
dropped, not guessed. Smallest answer — no cross-home identity inference in
the usage path.

### R-7 — What Phase D must not build

- No unified cross-home index, and no build-time index merging.
- No global score calibration across homes.
- No cross-home deduplication or content collapse (a future
  provenance-retaining collapse remains deferred per ADR 0116 R-3).
- No per-home fusion/weights/model configuration knobs.
- No cross-home consolidation or contradiction folds (R-4).

## Implementation seams (for the Phase D plan, not rulings)

- `packages/server/src/storage/local/backlog-service.ts` — the singleton
  service and module-level cache path become a per-home runtime factory.
- `packages/server/src/memory/bootstrap.ts` — composer, tracker, and
  `memory-usage.jsonl` path become per-home wiring; the JSONL moves to the
  home's `state/` for project homes.
- `packages/memory/src/search/orama-search-service.ts` — no fusion changes;
  gains per-home construction and (with ADR 0116 R-4) per-home readiness.
- The cross-home merger is new, small, and pure: ranks in, merged page out;
  it lives beside the search service, not inside Orama.

## Validation

- The cross-home cases already specified in ADR 0116 (size-skewed homes,
  duplicate local IDs, equal-rank tiers at the limit, zero-result homes,
  one degraded home beside one ready home, provenance stability) are the
  acceptance tests for R-3.
- The ADR 0116 fixture's memory-recall class runs against a project home
  with an overlay to prove R-5's containment: same stub shapes, same
  `usageFactor` behavior, zero repository dirtying (`git status` clean
  after a read is the test).
- ADR 0115's golden recall suite and the search golden suite must pass
  unchanged against a per-home runtime — their contracts are
  home-agnostic by construction.

## Consequences

Quartz builds Phase D against a fixed retrieval contract; onyx's stub and
usage work survives the overlay swap untouched; chert's fixture gains its
cross-home cases against a ruled merge; and the fusion question stays where
ADR 0116 put it — in the fixture gate, not in Phase D's lap.
