# 0113.1. Requirement Disclosure Surfaces

**Date**: 2026-07-16
**Status**: Proposed (design — beryl reviews before build)
**Thread**: child of ADR 0113 (user-defined substrates; operationalizes its Part 3 for the Requirement flagship — numbering confirmed with basalt). Imports ADR 0115's provenance-stub grammar and R-5 store-boundary law as seams; rides ADR 0114's one-retrieval-language fold.
**Driver**: Goga — *"during architecture work product features sometimes get derailed and lost. That is not acceptable."* (BRIEF-COMMON §9)

## TL;DR

Requirements exist so the human's product intent survives contact with
architecture work. That only happens if REQs are **pushed into the agent's
face at orientation time** and **carried as relations wherever decisions are
read** — not parked in a folder the agent never lists. Three rulings:

1. **R-1** — one **constraint-stub grammar**, minted at a single boundary
   (0115 R-5 extended to the Requirement substrate): id, title, status,
   compliance, and *assessment staleness*. A stale compliance check is the
   same disease 0115 treated — undeserved authority — so the stub carries
   `checked_days_ago` the way recall stubs carry `age_days`.
2. **R-2** — `backlog_wakeup` gains a bounded, scoped **constraints
   section**, ordered worst-first (violated → at_risk → unchecked →
   recently-satisfied), truncation stated as a count. Pure list fold; zero
   cost for projects with no REQs.
3. **R-3** — `backlog_get(id, context: true)` speaks **typed compliance
   relations**: a REQ exposes what it spawned and what violates it; an ADR
   exposes what it respects/violates — with compliance visible *on the
   stub*, so a violated REQ reads red without hydration.

Recall stays memory-only (R-4): constraints are pushed by wakeup and pulled
by search/get; stuffing REQs into the memory corpus would recreate the
two-dialect blur ADR 0114 just removed.

## Grounding (what exists today)

- 0113 Phase A (basalt's branch, `feat/adr-0113-user-defined-substrates`)
  ships `builtin:requirement@1` as a packaged definition
  (`packages/server/src/substrate-definitions/packaged-substrate-definitions.ts:47-66`)
  with identity `REQ-{key}` — but its canonical schema is only
  id/type/title/content with `additionalProperties: false`. **None of the
  0113-spec constraint fields (status, compliance, domain, checked_at/by,
  check_evidence, violated_by, grounds_in, spawned) survive a canonical
  write yet.**
- The claim/registry machinery (`core/substrates/claim-substrate-documents.ts`,
  `project-substrate-registry.ts`) exists on that branch; the open storage
  contract (0113 Phase B — REQ documents listable as runtime entities) does
  not exist anywhere yet.
- 0113 Part 3 already specifies the wakeup section's ordering and the
  omitted-count rule; this ADR turns that sketch into buildable contracts.
- 0115 R-5 (granite-ratified) requires provenance signals to reach read
  surfaces through one minting boundary; wakeup is a pure list fold by
  guardrail (beryl, 2026-07-16).

## R-1 — Constraint-stub grammar, minted once

A new `ConstraintStub`:

| field | source | note |
|---|---|---|
| `id`, `title` | identity/frontmatter | `REQ-0003`, human need |
| `domain?` | frontmatter | e.g. `fleet`, `aime` |
| `status` | frontmatter | intake/ruled/building/done/dropped |
| `compliance` | frontmatter | unchecked/satisfied/at_risk/violated/not_applicable — defaults to `unchecked` when absent |
| `checked_days_ago?` | `checked_at` | assessment staleness; absent = never assessed |
| `violations?` | `violated_by` | count + ids (≤3 shown), only when non-empty |

Minting lives in **one exported function** (working name
`toConstraintStub`, in the requirement disclosure module beside the
registry), the exact shape `toMemoryEntry` gives memories: no read surface
(wakeup, get, viewer) parses REQ frontmatter directly. Malformed dates
follow the 0115 single policy (normalize at the mint; staleness reads
"never assessed", not epoch). This is 0115 R-5 applied to the second
substrate — and the precedent for every future substrate that opts into
disclosure (0113 R7's `disclosure` descriptor points here).

Token cost: a full stub ≈ 25–35 tokens; the default 5-stub section ≈ 150
tokens inside wakeup's ~600-token budget — acceptable only because the
section is scoped and bounded (R-2), and absent entirely when a project has
no requirements.

## R-2 — Wakeup constraints section

`WakeupResult` gains `constraints: ConstraintStub[]` +
`metadata.constraints_omitted: number`:

- **Ordering (worst-first, per 0113 Part 3)**: violated → at_risk →
  unchecked-active → most-recently-checked satisfied. Within a band, most
  recently updated first. `dropped` and `not_applicable` never appear.
- **Bounds**: `max_constraints` param, default 5, 0 disables. When
  truncated, `constraints_omitted` states the count — the briefing never
  implies completeness (0113 Part 3 rule).
- **Scope**: the existing `scope` param filters constraints the same way it
  filters knowledge — REQs are per-home documents (0112), so the project
  wakeup shows the project's constraints.
- **Guardrails**: pure list fold (list REQ documents → mint → filter →
  sort → slice); no search dependency; R-5 containment via `toConstraintStub`.

## R-3 — Compliance as typed relations in `get(context: true)`

0114's `ContextStubs` grew generic `references`/`referenced_by` from URL
scraping. 0113 R6 makes relations typed data; compliance rides that:

- `get(REQ-0003, context: true)` → relation groups `spawned` (ADRs/tasks it
  produced), `violated_by`, and reverse `respected_by` (ADRs declaring
  `respects: [REQ-0003]`).
- `get(ADR-0117, context: true)` → groups `respects`, `violates`,
  `spawned_by` — each REQ stub carrying `compliance`, so a violated
  requirement is visible in the relation list itself.
- Violations render warning-first in tool output (the 0113 rule: human
  authority, visible warning, never a server-side veto).

Mechanically this generalizes `get-context`'s reverse-reference index from
"ID mentioned in a URL" to "typed relation field declared by the substrate
definition" — the relation names come from the registry, not hardcoded REQ
knowledge, so ADR/REQ are merely the first users.

## R-4 — What the ask-surfaces do NOT do

- **Recall stays memory-only.** REQs are constraints, not memories; they
  reach the agent via wakeup (push), search (pull — REQ documents index
  like any substrate under 0116/Phase C), and get-context (expand).
  Memories may cite REQ ids in `entity_refs`; the relation language, not
  corpus injection, is the bridge.
- **No compliance computation.** The server surfaces stored assessments; it
  never judges compliance itself (0113: requirements are human authority,
  not hidden policy code).

## R-5 — REQ field contract (input to basalt's Phase B packaged-schema uplift)

Basalt owns expanding `builtin:requirement@1`'s canonical schema; this is
the contract the disclosure surfaces consume. Field name = relation type
(0113 R6); relation values are arrays of canonical display ids.

| field | shape | write invariant |
|---|---|---|
| `status` | enum `intake\|ruled\|building\|done\|dropped` | defaults `intake` |
| `compliance` | enum `unchecked\|satisfied\|at_risk\|violated\|not_applicable` | defaults `unchecked`; canonical write of a value ≠ `unchecked` requires `checked_at` + `checked_by` |
| `domain` | string or string[] | optional |
| `date` | ISO date | intake date, optional |
| `uploaded_by` | string | optional |
| `grounds_in` | string[] | optional; free pointers (north-star anchors, MEMO-/PROMPT ids, URLs) |
| `spawned` | string[] (ids) | optional |
| `supersedes` | string[] (ids) | optional |
| `checked_at` | ISO date-time | optional |
| `checked_by` | string | optional |
| `check_evidence` | string[] | optional |
| `violated_by` | string[] (ids) | optional; canonical write rejects `compliance: satisfied` while `violated_by` is non-empty (a stored contradiction) — every other combination is human authority, untouched |

Reads stay lenient/lossless (0113 R4): all fields optional on read, unknown
extras preserved. The mint (R-1) is the only consumer that interprets them.

## Preconditions and phasing

1. **Schema**: basalt expands `builtin:requirement@1` to the R-5 contract
   in 0113 Phase B (confirmed owner; this ADR is the field spec).
2. **Listability**: R-2 needs REQ documents enumerable as runtime entities —
   the stated dependency is **0112's docs-native LocalRuntime/storage stack
   (quartz, bca510e lineage) composed with 0113 Phase B's registry-backed
   read/write path**. Design is final now; build starts when that seam
   lands.
3. **Build order once unblocked**: mint + stub (R-1, testable against raw
   frontmatter fixtures immediately) → wakeup section (R-2) → typed
   relations in get-context (R-3, coordinate with whatever 0113 Phase C
   ships for relation metadata).

## Falsifiability

- If real sessions show the constraints section ignored (violated REQs
  surfaced at wakeup but never opened or addressed), push harder on R-3
  (relations at decision-read time) and demote the wakeup section, not the
  reverse.
- If 5 stubs regularly truncate real projects (omitted count > 0 most
  wakeups), the bound is wrong — revisit before anyone habituates to
  incomplete briefings.
