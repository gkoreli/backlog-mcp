# 0106.2. Cleanup — Retire Task-as-Synonym-for-Entity; Introduce `<Type>Entity` Aliases

**Date**: 2026-06-17
**Status**: Accepted (execution delegated)
**Backlog Item**: TASK-0688
**Thread**: applies the ADR 0106.1 glossary; enables the ADR 0106 port vision

## Problem

`Task` is overloaded. It means two different things in the code, and the
collision is the source of the "Task-as-synonym-for-Entity rot":

1. **The task substrate's narrowed instance type** — legitimate:
   `export type Task = z.infer<typeof TaskSchema>` (`shared/substrates/task.ts:16`).
2. **A generic stand-in for "any entity"** — a fossil from when tasks were the
   only type. Evidence:
   - `storage/schema.ts`: `createTask` / `CreateTaskInput` build & return the
     discriminated `Entity` (folders, crons, milestones) — task-named, entity-typed.
   - `storage/backlog-service.ts`: imports `Entity`, but the storage class is
     `TaskStorage`, the field `taskStorage`, params `task: Entity`, locals
     `allTasks` — `Task` used to mean `Entity` throughout.
   - Test helpers: `function makeTask(o): Task` (9+ files) construct narrowed
     entities.

Per ADR 0106.1, **Substrate = definition, Entity = instance, Projection = response
view**, and "Task" must mean the task substrate *only*. The overload violates
that and makes the codebase read as if tasks are still privileged.

## Decision

Two coordinated changes, both glossary-driven (ADR 0106.1):

### 1. Retire `Task` as a generic synonym for `Entity`

Rename fossils that mean "any entity" to entity vocabulary:

| Current (fossil) | Target | Note |
|---|---|---|
| `createTask` (fn) | `createEntity` | returns `Entity`; defaults type to Task |
| `CreateTaskInput` | `CreateEntityInput` | derive from `z.input<EntitySchema>` (see §3) |
| `TaskStorage` (class) | `EntityStorage` | stores all entity types |
| `taskStorage` (field) | `entityStorage` | |
| `task: Entity` params | `entity: Entity` | |
| `allTasks`, `iterateTasks` | `allEntities`, `iterateEntities` | |
| `makeTask(): Task` (tests) | `makeEntity(): TaskEntity` | a narrowed task entity (see §2) |

**Do NOT rename** genuine task-substrate references (the `Task` `z.infer` alias
itself, `TaskSchema`, `TaskSubstrate`) — those correctly mean the task substrate.

### 2. Introduce `<Type>Entity` aliases — narrowed derivatives of the generic `Entity`

At the port/consumer level, a narrowed-per-type entity should *read* as "an Entity
of this substrate", not as a bare noun that collides with the substrate name.
Define them as derivatives of the discriminated union:

```ts
// shared/src/substrates/registry.ts (or a dedicated aliases module)
export type TaskEntity      = Extract<Entity, { type: 'task' }>;
export type EpicEntity      = Extract<Entity, { type: 'epic' }>;
export type FolderEntity    = Extract<Entity, { type: 'folder' }>;
export type ArtifactEntity  = Extract<Entity, { type: 'artifact' }>;
export type MilestoneEntity = Extract<Entity, { type: 'milestone' }>;
export type CronEntity      = Extract<Entity, { type: 'cron' }>;
export type MemoryEntity    = Extract<Entity, { type: 'memory' }>;
```

Rationale (maintainer direction): `TaskEntity` self-documents that it is the
generic `Entity` narrowed to one substrate — the derivative relationship is
explicit. It also frees the bare word `Task` from double duty.

**Migration of the existing bare aliases**: the current `export type Task =
z.infer<typeof TaskSchema>` becomes the `*Entity` form. Two acceptable paths —
the executor picks based on blast radius measured in step 0:
- **(a) Replace**: rename `Task`→`TaskEntity` (etc.) at definition and all call
  sites. Cleanest end-state; widest diff.
- **(b) Add + deprecate**: add `TaskEntity` as the canonical alias, keep `Task`
  as a `@deprecated` re-export for one release, migrate call sites incrementally.
  Lower risk; lingering synonym.

Default to **(a)** unless step-0 blast radius is large enough that (b) is safer.
Note `Extract<Entity, …>` is equivalent to the per-schema `z.infer` and keeps the
union as the single source of truth.

### 3. Fold in the `createEntity` factory cleanup (already specified)

While renaming `createTask`→`createEntity`, also:
- Derive `CreateEntityInput` from `z.input<typeof EntitySchema>` minus
  server-stamped fields (`created_at`/`updated_at`) with `type` optional.
- Delete the `if (x !== undefined)` ladder and the dead cross-type fields
  (`due_date/content_type/path` — reachable only by tests; `CreateParams` never
  carries them). A per-type-shaped input has no stray undefined cross-type keys
  to strip, so `EntitySchema.parse` handles everything.

This is safe and was validated at runtime in ADR 0106 (defaults apply;
cross-type keys rejected).

## Non-goals (explicitly out of scope for this refactor)

- **No new MCP intent-verbs** — that's ADR 0106, separate work. This is a rename +
  factory cleanup only; **no behavior changes, no API changes** to tool I/O.
- **No port-contract move to `shared`** — the `*Params/*Result` relocation is a
  later, separate decision.
- **No projection renames** — `RecallItem`, `SearchResultItem`, `ListItem`,
  `GetItem` are projections (ADR 0106.1) and stay as-is.

## Brutal critique / risks

1. **Blast radius.** `Task`/`task`/`makeTask` appear across storage, core, and
   ~50 test sites. Blind search/replace WILL break genuine task-substrate refs.
   **Mitigation**: step-0 audit classifies every hit (task substrate? generic
   entity? projection?) before editing; rename only the generic-entity class.
2. **`Extract<Entity, {type}>` vs `z.infer`.** Must produce identical types.
   **Mitigation**: typecheck is the gate; if any divergence, keep `z.infer` form
   and just rename the alias.
3. **Public re-exports.** `index.ts` re-exports `CreateTaskInput`/`createTask`
   (`server/src/index.ts:2`). Renaming is a breaking change to the package's
   public surface. **Mitigation**: if any external consumer exists, keep
   deprecated re-exports; otherwise rename outright (this is an internal monorepo).
4. **Churn vs. value.** Pure rename = large diff, no runtime change.
   **Mitigation**: do it in one focused, reviewable commit (or two: factory
   cleanup, then rename) so history stays legible; gate on full suite.

## Acceptance criteria (definition of done)

- [ ] `createTask`→`createEntity`; `CreateTaskInput`→`CreateEntityInput` derived
      from `z.input<EntitySchema>`; if-ladder + dead fields removed.
- [ ] `TaskStorage`→`EntityStorage`, `taskStorage`→`entityStorage`,
      `task: Entity`→`entity: Entity`, `allTasks`/`iterateTasks` renamed.
- [ ] `<Type>Entity` aliases exported from `shared`; bare-`Task`-as-entity call
      sites migrated; genuine task-substrate refs untouched.
- [ ] Test helpers `makeTask`→`makeEntity` returning `TaskEntity`.
- [ ] Projections (`*Item`/`*Result`) unchanged.
- [ ] `pnpm typecheck` clean; full server suite green (was 857 passing).
- [ ] No tool I/O or behavior change (verify create/update/remember manually).
- [ ] Commit trailers `X-AI-Prompt` / `X-AI-Tool` present.

## Execution & delegation

This is mechanical-but-wide and glossary-driven — a good delegation candidate.
The work is captured for an engineer agent in TASK-0688 with this ADR as the
spec. **Step 0 is mandatory**: produce the per-hit classification audit before
any edit, document it, then execute in reviewable chunks (factory cleanup →
storage rename → alias introduction → test helpers), running the suite between
chunks. Stop and escalate if blast radius exceeds the audit's estimate.
