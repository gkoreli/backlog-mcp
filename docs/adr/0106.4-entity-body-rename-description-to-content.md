# 0106.4. Rename the Entity Body Field `description` → `content`

**Date**: 2026-06-17
**Status**: Accepted (executed — TASK-0690, commits c828436..0936d2a)
**Thread**: continues 0106.1 (Substrate/Entity vocabulary) — closes the last body-field drift surfaced by the memory title/body work (TASK-0689)

## Context — two names for one concept

The markdown body of every entity is called **`description`** in the substrate/entity
world (`BaseEntitySchema.description`) but **`content`** in the memory plugin world
(`MemoryEntry.content`). `BacklogMemoryStore.toMemoryEntry` translates `description →
content` at the seam — the same two-definitions drift we just closed on `title`
(TASK-0689) and on the body's required-ness (commit 26e18cc).

The memory substrate's own design note already concedes the name is wrong:
> "The memory body (markdown description) IS the memory content."

`content` is the better name for a markdown body on **any** entity — "description"
implies metadata *about* a thing; for a memory (and arguably a task/artifact) the body
*is* the thing. Decision: rename universally so substrate and plugin finally agree, and
the seam stops translating.

## What the rename is NOT (de-risking — verified)

- **No SQL migration.** The D1 column is already named `body`
  (`d1-storage.ts:42` `body: string | null`), mapped to `entity.description` at
  `d1-storage.ts:79`. We remap `body ↔ entity.content`; the column name does not change.
- **No on-disk data migration.** Filesystem bodies are the **positional** gray-matter
  body (`matter.stringify(description, frontmatter)` — `filesystem-storage.ts:36`), not a
  keyed frontmatter field. Existing `tasks/*.md` files are untouched; only the in-memory
  property they hydrate into is renamed.
- **No read-path re-parse risk.** `markdownToEntity` casts `as Entity` without
  re-validating, so tightening/renaming the schema cannot break reads of existing data.

## Scope — the overloaded word `description`

`description` appears ~262× across ~73 files. It splits into two populations; **only the
first is renamed**:

### (a) Entity body — RENAME → `content`
- `BaseEntitySchema.description` (`shared/src/substrates/base.ts:42`) → `content`.
- The memory override `description: z.string().min(1)` (`memory.ts`) → `content: z.string().min(1)`.
- Every `entity.description` read (~61 property accesses) and `description:` entity-literal
  field (in core, storage, tools, cli, context stages, search indexing).
- Storage seams: `filesystem-storage.ts` destructure/stringify; `d1-storage.ts`
  `entity.description` ↔ `body` column mapping (column stays `body`).
- Search: `orama-schema.ts` / `orama-search-service.ts` / `snippets.ts` index the body.
- `BacklogMemoryStore.toMemoryEntry` — the translation `content: m.description` collapses
  to `content: m.content`; `MemoryEntry.content` is unchanged (it was already right).

### (b) NOT the entity body — LEAVE UNTOUCHED
- MCP tool-schema field descriptions: `.describe('…')`, `description:` inside Zod/JSON
  input schemas (`tools/*.ts`).
- OAuth client/scope `description` (`auth/oauth-*.ts`).
- `package.json` `description`.
- Any `description` that is documentation text, not an entity field.

**Blind search/replace is forbidden** — it corrupts population (b). Classify every hit
first (same discipline as ADR 0106.2's Step-0 audit).

## Decision on the public MCP/CLI input field

`backlog_create` / `backlog edit` currently accept a `description` input. Two options:

1. **Rename the input to `content` too** — full vocabulary consistency, but a tool-I/O
   change for agents/users.
2. **Keep the input named `description`, map to `content` internally** — zero I/O change,
   but reintroduces a (tiny, single-site) name translation at the port.

**Decision: rename the input to `content` as well (Option 1).** This is a
single-consumer personal tool; consistency end-to-end is worth the one-time change, and a
half-rename that leaves the public field as `description` defeats the purpose. Update tool
input schemas, CLI flags/options, their `.describe()` help text, and any docs/examples.
Verify create/edit/update manually post-rename.

> If an external consumer is later discovered, add a backward-compat alias at the port
> (accept both `content` and legacy `description`), not in the core/substrate.

## Consequences

**Positive**
- One vocabulary: substrate `content`, plugin `MemoryEntry.content`, MCP input `content`.
- `toMemoryEntry` no longer translates the body field — the seam shrinks.
- "content" is a more honest name for a markdown body on every entity type.

**Negative / risks**
- Large mechanical diff (~61 reads + ~128 literals + tool/CLI surface). Mitigation:
  typecheck + full suite are the gate; classification audit prevents population-(b)
  corruption.
- Public tool field rename. Mitigation: single-consumer tool; documented; manual verify.
