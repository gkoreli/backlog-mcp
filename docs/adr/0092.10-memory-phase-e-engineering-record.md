# 0092.10. Agentic Memory — Phase E Engineering Record

**Status**: Accepted — Phase E shipped and manually validated 2026-06-11
**Implements**: ADR 0092.9 rulings R-13…R-16
**Commits**: `7731f54` (research+plan ADR), `07de7fb` (implementation)

## North star check-in

**All five capabilities of the original ADR 0092 vision are now live**:
score decay, wake-up context, implicit capture, consolidation with enforced
provenance, and — with this phase — the echo/fizzle feedback loop. Memory
now *gets better with use*: useful memories rise, idle ones sink (bounded,
never hidden), every event is auditable, and the recall-demand log closes
the last open signal from Phase D.

## What shipped

- **`memory/usage-signal.ts`** — pure bounded multiplier (R-15): 14-day
  grace → neutral; then `0.3 + 1.2·(min(count,8)/8)·2^(−idleDays/30)`,
  clamped [0.3, 1.5]. Saturating count (Postgres/Redis precedent); idle
  decay is MemoryBank's spacing effect; reorders, never hides (Mem0's
  contract, Hindsight's ±10% philosophy).
- **`memory/usage-tracker.ts`** — two-tier recording after relatime/lazytime
  (R-13): JSONL audit log (`memory-usage.jsonl`, batched per event,
  replayable — and per R-16 it doubles as the recall-demand log Phase D
  deferred) + frontmatter summary (`usage_count`, `last_used_at`) flushed
  only on Fibonacci bucket boundaries or >24h staleness. Strong events only
  (R-14): expand (`backlog_get` on MEMO-) and cite (MEMO- ids in remembered
  content/refs) bump counters; stub appearances are logged but free —
  recall stays a pure read. Tracker failures never propagate.
- **Wiring**: `BacklogMemoryStore.recall` applies the factor over the full
  filtered candidate set before truncation (widened-pool lesson); hooks on
  `backlog_get`/`backlog_recall`/`backlog_remember` (MCP + CLI) via
  `ToolDeps.usageTracker`; Node wires the default, Worker omits (ADR 0104).

## Manual validation (maintainer's machine, real processes)

Two procedural memories, backdated past the grace period, then the loop:

- ✅ Recall logged a batched demand line; **no entity writes** — pure read.
- ✅ First expand of `MEMO-0002` → JSONL event + frontmatter flush
  (`usage_count: 1`, bucket 1).
- ✅ **One use did not flip the ranking** — correct: with a 2-doc corpus,
  rank normalization quantizes bases to 1.0/0.5, and a 0.45-vs-0.3 factor
  must not overcome a 2× relevance gap. Relevance dominates; usage biases.
- ✅ **Eight expands flipped it**: the heavily-used memory overtook the
  unused one (0.412 vs 0.360) — demonstrated usefulness reorders.
- ✅ **The relatime gate held**: 8 expands → only 3 frontmatter writes
  (buckets 1, 2, 3); all 11 events in the JSONL. Exact history without git
  churn — the strictatime→relatime trade, observed live.

## Distilled insights

- **Rank normalization quantizes small corpora** — with n candidates the
  base scores step by 1/n, so the usage band (5× spread) flips neighbors
  but not 2× rank gaps. This is the right default (relevance first), worth
  remembering when judging "why didn't my used memory move."
- **The JSONL/frontmatter split earns both audiences**: agents and tooling
  replay the exact event stream; humans see an honest summary in the file
  and the viewer. Neither surface lies; only the summary is rate-limited.
- **Validation infrastructure note**: the session's sandbox died (disk
  full) mid-phase; validation completed via Desktop Commander on the
  maintainer's machine — where, additionally, the full hybrid-search suite
  passes (local embeddings), confirming the two sandbox-skipped tests.
