# 0013.7. Transport, Bridge & Hosting — Decision Framework

> *Thread head [0013](./0013-http-mcp-server-architecture.md). This is the latest
> entry in the transport & bridge thread (0013 → 0013.1 … 0013.7); renumbered from
> 0103 on 2026-06-04.*

> **Why this ADR exists:** the question *"do I still need `mcp-remote`?"* has been
> reopened repeatedly (see thread 0013 → 0013.1 [stdio bridge], 0013.2 [mcp-remote],
> 0013.3 [HTTP migration], 0013.4 [streamable, blocked], 0013.5 [dual-mode, rejected],
> 0013.6 [streamable w/ current arch]). Each time it feels novel because the reasoning
> lived only in conversation, never on disk. This ADR captures the framework so the
> loop **stops reopening**. When the question comes up again: read this, don't
> re-derive it.

## Context

backlog-mcp can be reached by different MCP clients, and we kept conflating two
**independent** axes, which is what made `mcp-remote` feel mandatory (or useless,
depending on the day):

- **Axis 1 — where the server/state lives:** local daemon (`~/backlog/*.md`) vs.
  remote hosted server.
- **Axis 2 — how a client connects:** by `url` (HTTP/Streamable) or by `command` (stdio).

`mcp-remote` only ever operates on **Axis 2**. It is a stdio→HTTP adapter. "Local"
does **not** imply "stdio" — a local daemon is still an HTTP server on `localhost`.

### What `mcp-remote` is actually for

Per its author (geelen/mcp-remote), its stated purpose is:

> *"Connect an MCP Client that only supports local (stdio) servers to a **Remote**
> MCP Server, **with auth support**."*

The headline feature is **OAuth** — token storage (`~/.mcp-auth`), a callback server,
session isolation, dynamic client registration. The README explicitly calls it a
**stopgap** to be removed *"once clients natively support authorized remote servers."*

### How we invoke it today (`packages/server/src/cli/bridge.ts`)

```js
spawn(mcpRemotePath, [serverUrl, '--allow-http', '--transport', 'http-only'])
//                     ↑ http://localhost:3030  ↑ no auth      ↑ negotiation off
```

We point it at **localhost**, with **no auth**, with **transport negotiation disabled**.
That is the one configuration where *both* of `mcp-remote`'s reasons to exist
("remote" and "auth") are absent — we use ~10% of it (the bare stdio↔HTTP pipe) and
inherit 100% of its surface, including its hang-on-error behaviour, which we work
around by scraping stderr for `ECONNREFUSED`/`fetch failed` and force-killing it
(`bridge.ts:36`).

## Decisions

### D1 — The daemon model stays. (Affirmed)

We do **not** want N stdio clients to each spawn an in-process server (ADR-0013.5
"native dual-mode" Option 1). That reintroduces the exact problems ADR-0013 killed:
ID races in `getMaxId()` (filename-scan collisions), divergent in-memory Orama
indexes, and no shared viewer/event stream. `server-manager.ts:ensureServer`
deduplicates to **one** shared daemon; thin per-client bridges pipe stdio to it.

> This **supersedes the recommendation** of ADR-0013.5 (native `--stdio` in-process
> mode) for the multi-client case. In-process stdio is acceptable only for a
> genuinely single-client-at-a-time use, which we do not assume.

### D2 — `mcp-remote` is correct **by destination**, not globally. (Accepted)

The bridge-vs-no-bridge and mcp-remote-vs-native question resolves by what the
client is and where it points — **not** by taste:

| Client → target | Bridge? | Use `mcp-remote`? | Why |
|---|---|---|---|
| `url` client (e.g. Kiro CLI) → local **or** remote | ❌ none | ❌ | Native remote MCP (`RemoteMcpServerConfig`, incl. OAuth). The stopgap is already obsolete for it. Just set the `url`. |
| stdio-only client → **local** daemon (no auth) | ✅ thin | ❌ → native bridge | We use none of mcp-remote's OAuth and actively fight its error handling. A native `StdioServerTransport ↔ StreamableHTTPClientTransport` pipe is cleaner here. |
| stdio-only client → **remote authenticated** server (OAuth) | ✅ | ✅ **keep `mcp-remote`** | This is *literally* what it's built for. Reimplementing = rebuilding OAuth + token store + callback server. Do **not** hand-roll this. |

Corollary: Kiro CLI takes a `url` natively (confirmed: `RemoteMcpServerConfig` with
`url`, `headers`, OAuth, streamable HTTP). For Kiro, **local vs remote is just a
different URL** — no bridge either way.

### D3 — A native thin bridge is a *quality* swap, not a fix. (Accepted, low priority)

If/when we replace `mcp-remote` on the **local no-auth** path, it must be a
**transparent transport pair** — forward `JSONRPCMessage` frames between
`StdioServerTransport` and `StreamableHTTPClientTransport`, and **never** wrap it in
`McpServer`/`Client`. Wrapping it in a Client is precisely the bug that sank our
first custom bridge (ADR-0013.2: *"added a protocol layer instead of being a
transparent proxy"*). Any native bridge MUST carry an ADR-0013.2 guardrail comment and
a round-trip frame test. This is tidiness (deletes the stderr hang-hack), not a
capability gain — never let it block or precede higher-leverage work.

## Open Tension — Hosting: Workers+D1 vs. a filesystem-native host

**Status: OPEN. Not decided here. Recorded so we stop rediscovering it.**

backlog-mcp is, at its core, a **filesystem/markdown tool**: `task-storage.ts`
stores tasks as `TASK-NNNN.md` with gray-matter frontmatter, lists via `readdirSync`,
and allocates IDs by regex-scanning filenames. Human-readable, git-diffable,
grep-able files are the product's soul.

Cloudflare Workers **has no filesystem**, which forced a parallel SQL universe:
`d1-adapter.ts` / `d1-backlog-service.ts` / `worker-entry.ts` alongside the
filesystem stack. The tension: we contort a file-shaped product to fit a runtime
that has no files.

**The honest counterweight (do not ignore):** Workers+D1 is **already shipped and
hardened**, not a sketch — ADR-0089 (migration), 0091 (clean worker bundle), 0093
(D1 production hardening), and **0100 (a real GitHub-OAuth/PKCE/JWT auth system)**
serving a live endpoint to Claude.ai / Claude Desktop / ChatGPT. Walking that back
has real cost.

A filesystem-native host (e.g. **Fly.io Machine + persistent Volume** mounted at
`BACKLOG_DATA_DIR`) would let the *existing* `filesystem-adapter` run unchanged,
delete the D1 fork, and still satisfy ADR-0013's "hosted, persistent, multi-client"
vision — because **"remote MCP" only ever required HTTP transport, never Workers.**
Constraint if pursued: **one** Machine + **one** Volume (single writer), or we
re-create ADR-0013's "storage duplication across processes."

**Trade to resolve when revisited:**
- Workers+D1: scale-to-zero, free edge, zero-ops, **but** permanent forked SQL
  storage layer + a product that can't show its own files.
- Fly (or any disk host): filesystem-native, mostly *deletes* code, single storage
  universe, **but** loses edge/scale-to-zero and is an always-on box to run, and
  means migrating/retiring the shipped 0089/0093/0100 investment.

This is a **direction lean, not a commitment.** Revisit deliberately as its own ADR;
do not let it leak into unrelated changes.

## Consequences

- The "do I need mcp-remote?" question now has a written answer (D2). **Point at this
  table instead of re-deriving.**
- No code changes are mandated by this ADR. `mcp-remote` stays as-is on the local
  path until/unless we choose the D3 swap.
- ADR-0013.5's native-dual-mode recommendation is superseded for the multi-client case
  (D1). ADR-0013.2's transparent-proxy lesson is reaffirmed as a hard constraint (D3).
- The hosting question is parked as an explicit Open Tension, not silently dropped.
