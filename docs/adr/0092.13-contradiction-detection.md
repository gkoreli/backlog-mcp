# 0092.13. Contradiction Detection

**Status**: Accepted ‚Äî shipped 2026-06-16 (`08f013c`). Verified: 9/9
core-contradictions tests green (live-pair flag; single-holder, expired,
and no-key negatives; most-recent ordering; service counts; per-memory
`contradictsFor`). Server typecheck clean; viewer typecheck clean, 96
tests green; viewer-routes green. The wider suite's search/orama +
`sharp` failures are the sandbox's macOS‚ÜíLinux native-binary mismatch,
not regressions. Live-viewer visual spot-check of the contradiction chip:
recommended next time the server runs.

**As-built note**: the viewer surface is the REST detail pane
(`/tasks/:id` ‚Üí `DocumentView` ‚Üí `MetadataCard`), where the computed
`contradicts` field lives ‚Äî not the `.md` resource-viewer pane, which
reads stored frontmatter and so never carries a *derived* field. The
key has no underscore, so it survives `DocumentView`'s `_`‚Üíspace
transform that the `valid_until`/`derived` chips (0092.11) only dodge
because they render through the resource-viewer path. A follow-up could
unify the two panes' chip rendering; logged, not silently shipped.

**Why now**: ADR 0092.11 made memory *legible* in the viewer (evidence
chains, validity/inference chips) but explicitly deferred the one R-9
piece that needs a server: contradiction surfacing. This ADR builds the
detection and wires it to **both halves of the north star** at once ‚Äî
*visibility* in the viewer, *control* through the agent and CLI. It is
the human-in-the-loop differentiator: no surveyed competitor (ADR 0092.5)
shows an agent its own contradictory beliefs and asks a human to adjudicate.

**Evidence basis** (0092.5 R-9): *"Contradiction surfacing is a viewer
feature, not a write-path feature. Deterministic detection (shared
`state_key` with overlapping validity, or near-duplicate embeddings with
divergent content) flags pairs for human adjudication. Resolution is
never automatic."* ‚Äî MemPalace's 13-months-unshipped lesson: automatic
belief-revision is where memory systems die. We flag; the human decides.

## The signal we already own

ADR 0092.5 R-2 (verified in `memory-store-contract.test.ts`): writing a
memory with a `state_key` soft-expires *every live previous holder* of
that key. So under the write path's own invariant there is **at most one
live holder per `state_key`**. Two live holders of one key is therefore
not a normal state ‚Äî it is the invariant breached: a direct store write,
an import, or a bug bypassing `remember`. That makes the deterministic
detector trivial *and* meaningful: **group live, non-expired memories by
`state_key`; any group of ‚â•2 is a contradiction set.** No embeddings, no
thresholds, no cosine ‚Äî a pure, replayable fold over data we already
keep. (The near-duplicate-embedding signal in R-9 is a second, fuzzier
detector that needs the search stack; deferred ‚Äî see below.)

## Design

- **Pure core**: `groupByStateKey(memories, {now})` ‚Üí `ContradictionGroup[]`.
  Considers only *live* memories (`valid_until` absent or in the future);
  buckets by `state_key`; keeps buckets with ‚â•2 members. Each group
  carries the members' evidence (id, title, `created_at`, `valid_until`,
  `entity_refs`, `source`) so the human adjudicates with full context,
  never just two ids. Deterministic ordering: most-recent contradiction
  first, then by key.
- **Service edge**: `detectContradictions(service)` lists `Memory`
  entities and folds ‚Äî same shape as `consolidationCandidates` (ADR
  0092.7). No new IO, no injected readers; the substrate is the source.
- **Control via agent + CLI** (the verbs that *act*):
  - MCP tool `backlog_contradictions` ‚Äî the agent can detect its own
    conflicts mid-task and resolve them with the existing `remember`
    (`supersedes`/`state_key`) or `forget` verbs. Detection here, action
    through the verbs already shipped ‚Äî R-9's "never automatic" holds.
  - CLI `backlog contradictions` ‚Äî the human runs the same fold,
    rendering each group with member ids and dates.
- **Visibility via viewer** (reusing 0092.11's generic-renderer win):
  the memory detail response gains a computed `contradicts: [MEMO-‚Ä¶]`
  field listing the *other* live holders of its `state_key`. The
  existing `MetadataCard` already turns entity-id arrays into split-pane
  links ‚Äî so the evidence is navigable for free. One new chip case
  (`contradicts` ‚Üí a red "contradiction" chip) makes it unmissable. No
  bespoke panel, no new fetch path: the same "one renderer, whole-system
  win" lever 0092.11 pulled.

Resolution is always a human/agent decision through `remember`/`forget`.
The detector has no write path ‚Äî it cannot revise a belief, only show one.

## Deferred (next loops)

- **Near-duplicate-embedding contradictions** (R-9's second detector):
  divergent content under *different* keys, caught by cosine ‚â• œÑ on
  embeddings. Needs the orama/search stack and a divergence threshold ‚Äî
  a fuzzier signal with false positives, worth its own loop once the
  deterministic one is proven in the viewer.
- **A standalone contradictions dashboard** in the viewer (all groups at
  once). The per-memory chip ships first; the roll-up follows demand.
