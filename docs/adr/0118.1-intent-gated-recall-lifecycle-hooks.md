# 0118.1. Proactive Recall Refresh — Intent-Gated Recall at Lifecycle Boundaries

**Status**: Proposed — design only; no implementation authorized by this ADR.

## Decision in one sentence

Put a small recall decision rubric in the session-start orientation, let the
agent call recall only when its current intent warrants it, return bounded
provenance-bearing stubs, and use compaction hooks only for lifecycle receipts
and one canonical re-prime — never for automatic memory injection.

## Why ADR 0118 needs a refresh

ADR 0118 correctly identified three durable facts:

1. memory that is never recalled has no practical value;
2. recall must be cheap, fail-open, and stub-first; and
3. hook mechanics belong to the client harness, while backlog-mcp owns the
   transport-free recall contract.

Its operational premise is now dated. It concluded that shipping memory
systems recalled on every user prompt, so removing the decision from the agent
was the only dependable answer. The verified July 2026 landscape in the
[uplift idea garden](../proposals/uplift-idea-garden-2026-07.md)
shows a different convergence:

- Mem0's current Claude Code plugin injects a **decision rubric once at session
  start**; the agent decides when and how to search rather than receiving
  memories on every response ([Mem0](https://mem0.ai/blog/claude-code-memory)).
- claude-mem uses progressive disclosure: search returns a compact index of
  IDs, then the caller opens only selected observations
  ([claude-mem](https://github.com/thedotmack/claude-mem)).
- MemPalace provides session-boundary wake-up and pre-compression lifecycle
  mounts rather than proving that every ordinary prompt needs injected recall.

This matches backlog-mcp's own North Star better than 0118's original design:

- Tenet 2: never dump; disclose progressively.
- Tenet 8: every always-present surface pays a permanent context cost.
- Tenet 9: build the smallest answer at the pressure point.
- Invariant 8: do not act on human-authored material uninvited. The direct
  invariant governs source mutation; its session-context corollary here is
  equally important: do not silently augment every human prompt with retrieved
  material the human did not request and the agent did not choose.

The pressure is still real: agents sometimes fail to recall relevant prior
knowledge. The smallest answer is to improve the decision available at the
natural orientation boundary, not to tax every turn forever.

## Decision

### R1. “Proactive” means available before a miss, not injected before every turn

The recalled **results** remain pull-based. What becomes proactive is the
bounded instruction that tells the agent when recall is valuable and how to do
it cheaply.

There is no `UserPromptSubmit` recall hook. An ordinary prompt causes:

- zero recall requests;
- zero recall-result tokens added to context; and
- zero memory writes.

This supersedes ADR 0118's unconditional per-turn hook, its “none gate,” and
its claim that the recall coin-flip is eliminated by infrastructure. The agent
still makes the intent decision. The design makes that decision legible and
cheap instead of trying to remove it.

### R2. Session start carries one small decision rubric, not recalled memories

`backlog-mcp wakeup` is the existing orientation surface. Its future bounded
output may include a rubric equivalent to:

> Recall before reconstructing project-specific knowledge likely learned in a
> prior session: conventions, decisions, previous fixes, preferences, or
> corrections. Skip recall for generic knowledge, facts already visible in the
> current files, or when the user asks you not to. Search stubs first; open only
> the IDs that matter.

This text is policy, not retrieved data. It should fit in roughly 50–100 tokens,
appear once in the canonical session orientation, and introduce no new tool.
The existing `recall` and `get` verbs remain the two-stage path established by
ADR 0092.5 and strengthened by ADR 0115:

```text
session start -> bounded wakeup + decision rubric
ordinary turn -> agent decides whether prior project knowledge matters
no            -> continue with no recall cost
yes           -> recall(query) -> provenance-bearing stubs
                              -> get(selected IDs) only when needed
```

If another client owns the session orientation, backlog-mcp does not inject a
second wakeup. The owner composes the rubric into its one bounded orientation
payload. Aime ADR 0032 proposes this ownership pattern; it is a sibling design,
not current behavior.

### R3. Claude Code mounts at SessionStart and compaction boundaries

Claude Code is the first concrete harness because its lifecycle is explicit:
`SessionStart` distinguishes `startup`, `resume`, `clear`, and `compact`, while
`PreCompact` and `PostCompact` distinguish `manual` and `auto` compaction. Its
official hook contract adds command stdout to session context and recommends
keeping session-start hooks fast
([Claude Code hooks](https://code.claude.com/docs/en/hooks)).

#### SessionStart: one canonical orientation

After Slice A adds the rubric and a hard total wakeup ceiling, a project that
does not already have a lifecycle owner can mount the CLI directly:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume|clear|compact",
        "hooks": [
          {
            "type": "command",
            "command": "env",
            "args": [
              "BACKLOG_DOCS_NATIVE=1",
              "backlog-mcp",
              "--home", "project",
              "--project-root", "${CLAUDE_PROJECT_DIR}",
              "wakeup",
              "--max-knowledge", "0",
              "--max-constraints", "3",
              "--max-completions", "3",
              "--max-activity", "3"
            ],
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

The project root is explicit because the detached daemon must never infer
caller scope from its own working directory. `BACKLOG_DOCS_NATIVE=1` is the
current migration gate and disappears from this recipe after the Phase E
cutover. `--max-knowledge 0` is load-bearing: today's wakeup otherwise includes
semantic/procedural memory stubs, which would contradict the decision to inject
policy rather than retrieved memory. A command hook is intentional: Claude
Code documents that `SessionStart` often fires before MCP servers connect, so
an MCP-tool hook is not reliable at this boundary.

The command must be fail-open and bounded. The hook is not ready to install
before Slice A: current wakeup limits knowledge, constraints, completions, and
activity, but not the active-task and current-epic sections. Slice A supplies a
hard total ceiling with the rubric as non-droppable content, without adding a
new caller knob. Failure produces no fabricated context and must not block the
session. Clients that perform a canonical re-prime omit this hook and include
the rubric in that existing payload instead.

#### PreCompact: receipt or already-authorized checkpoint only

`PreCompact` is **not** a recall trigger and **not** permission to create a
memory. A generic backlog-mcp integration does nothing at this boundary.

A harness that already owns continuity may attach a client-owned command hook
whose only generic responsibility is to record a typed lifecycle receipt such
as `{ session_id, trigger, phase: "pre-compact" }`. It emits no stdout and does
not copy transcript content into memory. If the user or harness separately
authorized a checkpoint, the client may flush that already-authorized work;
the hook event itself supplies no new authority.

#### PostCompact and SessionStart(source=compact): receipt, then re-prime

`PostCompact` may record the matching effect receipt. It does not inject recall
results or rewrite Claude Code's generated compact summary. The following
`SessionStart` with source `compact` is where the lifecycle owner emits one
bounded canonical re-prime.

This follows the proposed sibling pattern in
[Aime ADR 0032](https://github.com/gkoreli/aime/blob/main/docs/adr/0032-context-lifecycle-management.md):
PreCompact records intent, PostCompact records effect, and SessionStart owns
one bounded re-prime. ADR 0032 is design-only at the time of this decision.
Aime remains a client; backlog-mcp does not depend on it or duplicate its
active-goal anchor.

Other harnesses should map the same three semantics, not copy event names:

| Semantic boundary | Client responsibility | Memory payload |
|---|---|---|
| Session begins or resumes | One bounded orientation + recall rubric | No recalled results |
| Compaction is about to occur | Optional lifecycle receipt; flush only prior authorization | None |
| Compaction completed | Optional effect receipt, then one canonical re-prime | No automatic recall |

Kiro-specific recipes in ADR 0118 are historical evidence, not a current
implementation target. A new Kiro recipe requires fresh contract verification.

### R4. The optional HTTP adapter is explicit, stub-only, and private

Clients that cannot use MCP may eventually call a narrow warm-daemon adapter:

```http
POST /memory/recall
Authorization: Bearer <token>
Content-Type: application/json

{
  "query": "the project-specific fact I am trying to recover",
  "context": "optional focal entity ID"
}
```

ADR 0118 considered a `GET` route and rejected a separate REST adapter as
duplicative. This refresh reverses that disposition only for a real seam:
clients that cannot use MCP. It chooses `POST` because recall queries can
contain human prompt and project context; they must not be placed in URLs,
browser history, proxy cache keys, or routine access logs.

The route is an adapter over the existing `core/recall.ts`, not a second search
implementation and not a new semantic tool. Its v1 contract is deliberately
fixed:

- nonblank `query`, at most 2,000 characters;
- optional nonblank `context`, at most 200 characters;
- `full: false` always;
- at most five items and the core's existing greedy token packing with an
  approximate 500-token target (the first stub is retained even if it exceeds
  the target);
- all persisted memory layers, with no caller-supplied layer/tag/score knobs;
- existing `X-Backlog-Home` / `X-Backlog-Project-Root` selection semantics,
  including one home only and headers taking precedence;
- the existing `RecallResult` envelope plus home provenance;
- `200` with an empty item list when no composer or no match exists;
- `400` for malformed input or invalid home selection;
- `500` for a genuine runtime failure; and
- `Cache-Control: no-store` on every response.

The adapter records the existing `recordRecall(query, returnedIds)` signal
once, after results return. Unlike the current MCP wrapper, the HTTP adapter
must guard this derived write so a tracking failure never changes the recall
response.

The server never decides whether the query deserves recall. Calling this route
is the caller's intent signal. There is no intent classifier, prompt capture,
or automatic forwarding from a per-turn hook.

#### Privacy gate

The current Node server listens on `0.0.0.0`, while REST routes outside `/mcp`
do not all share MCP bearer authentication. Therefore `/memory/recall` must not
be registered as an unauthenticated route on that listener.

Implementation is gated on one of these already-understood protections:

1. the route shares the `/mcp` bearer-auth boundary **and** an API key or JWT
   secret is configured; or
2. the enclosing server is explicitly loopback-only.

This ADR does not broaden the listener, mint a recall-specific secret, or
declare localhost alone to be authentication. Until the gate is met, clients
use the existing authenticated MCP call or the local CLI.

### R5. Fail open at the harness; fail closed at scope and privacy boundaries

A recall outage must not stop an agent from answering. SessionStart command
failure and explicit recall transport failure are visible diagnostics followed
by normal agent work.

Scope and privacy failures are different. Invalid home selection, missing
project identity, failed authentication, and ambiguous cross-home requests
return errors without falling back to another home. “Fail open” never means
“search a broader store.”

## Explicit non-goals

This refresh does **not** introduce:

- `UserPromptSubmit` or any other per-turn recall hook;
- unrequested injection of recall results;
- an LLM or heuristic intent classifier in backlog-mcp;
- automatic memory creation at PreCompact, PostCompact, SessionEnd, or Stop;
- persistence of recall results or Claude Code compact summaries;
- a new MCP tool, retrieval engine, index, scoring formula, or substrate;
- caller-controlled full-body recall through the HTTP adapter;
- cross-home recall or silent fallback between project and global homes;
- Aime-specific behavior in backlog-mcp core;
- a Kiro recipe without current primary-source verification; or
- D1/Worker parity before a real remote client creates that pressure.

These are not deferred features hidden inside the design. They are outside the
decision. Tenet 8's context-cost law and Tenet 9's pressure rule require new
evidence before any is reconsidered.

## Smallest implementation slices, if separately authorized

### Slice A — session-start policy (S)

1. Add the bounded decision rubric to `wakeup` output.
2. Enforce the North Star's existing wakeup budget as a hard total ceiling,
   preserving the rubric and an honest truncation marker before optional
   detail; do not add a configuration knob.
3. Document the Claude Code `SessionStart` recipe with knowledge recall
   disabled.
4. Test that the rubric appears once, no knowledge stubs are injected, and the
   total output stays within the ceiling even with many active tasks and epics.

This slice proves the behavioral hypothesis without server surface area.

### Slice B — protected HTTP adapter (S–M)

Only if a real non-MCP client needs it:

1. satisfy the privacy gate;
2. add the fixed `POST /memory/recall` adapter over `core/recall.ts`;
3. reuse home selection, provenance, and usage tracking; and
4. test input bounds, stub-only output, empty results, authentication, and
   cross-home failure.

Slice B is not a prerequisite for Slice A. The CLI and authenticated MCP path
already exist.

## Acceptance and falsification

The design succeeds when:

1. ordinary user prompts produce no recall request and no injected recall
   payload;
2. session-start orientation contains the rubric once and remains bounded;
3. PreCompact and PostCompact produce no recall payload and no uninvited memory
   mutation;
4. an explicit recall returns no more than five provenance-bearing stubs and
   never returns bodies;
5. only selected IDs are hydrated through the existing `get` path;
6. invalid or ambiguous home selection fails without searching another home;
7. the HTTP route cannot expose recall over an unauthenticated non-loopback
   listener; and
8. recall failure does not block the agent's primary task.

Reconsider automatic hooks only after measured sessions show that the rubric
still causes materially harmful recall misses, and compare that miss rate
against the context noise and latency of automatic injection. Anecdote alone
does not reopen the design.

## Consequences

### Positive

- Zero per-turn context or process tax.
- The existing four-verb memory language remains intact.
- Recall stays progressive: rubric, stubs, then selected bodies.
- Claude Code, Aime, and future harnesses share lifecycle semantics without
  creating a framework-specific dependency in core.
- Compaction integration preserves human and client authority instead of
  turning a lifecycle event into implicit permission to write memory.
- A future HTTP path is useful to thin clients without exposing prompt-derived
  queries through URLs or an unauthenticated listener.

### Trade-offs

- Intent-gated recall can still be skipped. This ADR accepts that residual miss
  rate until evidence shows it is worse than an every-turn tax.
- Clients with their own re-prime composer must include the rubric themselves;
  two independent orientation hooks would duplicate context.
- The HTTP adapter remains unavailable to clients that have neither bearer
  authentication nor a loopback-only server. That is a truthful security
  boundary, not a setup convenience to bypass.

## Evidence retained from ADR 0118

ADR 0118's June 2026 measurements — 38 ms for warm MCP recall and 109 ms for its
then-current hook path — remain useful feasibility evidence, not a current
performance guarantee. The transport-free core, stub-only default, fail-open
client posture, and warm-daemon compatibility all survive this refresh.

What does not survive is the inference that low latency makes every-turn
injection free. Tenet 8 counts context and attention, not only milliseconds.
