# 0092.4. Agentic Memory — Phase A + B Engineering Record

**Status**: Accepted — Phases A and B of ADR 0092.3 shipped 2026-06-10
**Implements**: ADR 0092.3 (memory experience & substrate), ADR 0083 (search fixes)
**Commits**: `1cc4f0f` (ADR 0092.3), `907a4b1` (Phase A), `ff2a40b` (Phase B)
**Verification**: 809 tests green (784 server + 25 memory), typecheck clean in shared/memory/server/viewer

## The north star (restated)

> An agent using backlog-mcp should be measurably smarter in week 10 than in
> week 1 — on the same project, because of memory.

Phase A made recall *correct*; Phase B made memory *durable and first-class*.
Together they close the two structural holes that made the Phase 1–3 memory
system a demo rather than a memory: recall that couldn't be trusted, and
memories that couldn't survive a process exit.

## What shipped

### Phase A — search foundation (ADR 0083 criticals / TASK-0676)

All seven "immediate fixes" plus two of the four "architecture evolution"
items from ADR 0083, thirteen months after they were specified:

- **Rank normalization replaces MinMax in fusion** (`scoring.ts:rankNormalize`).
  MinMax mapped the lowest BM25 scorer to literally 0.0 — when the *relevant*
  document was the lowest scorer (query terms buried in compound words), it
  became invisible to every downstream modifier. Rank normalization scores by
  position with shared-tie semantics; the lowest scorer keeps `1/n > 0`.
- **Exact title-phrase pin** (`scoring.ts:applyExactTitlePin`, final pipeline
  stage). All query tokens contiguous in the tokenized title → +2.0 offset,
  clearing the un-pinned band (fusion ≤1.0 + coordination ≤0.8) while
  preserving relative order among pinned docs. Single-token queries pin only
  whole-title matches — one common word must not pin half the corpus.
  Navigational queries ("backlog mcp") now beat decay and coordination noise.
- **`id` removed from TEXT_PROPERTIES and the default boost.** Every ID
  tokenizes to its type prefix ("TASK-0009" → "task"), so the 10× id boost
  made any query containing "task"/"epic" match every document of that type.
  ID-shaped queries now short-circuit through the intent parser — including
  in the task-only `search()` method, which previously had no intent routing.
- **Correctness bugs**: awaited `insert`/`insertMultiple` in `index()` and
  `indexResources()` (silently-swallowed failures); atomic
  `updateDocument`/`updateResource` with previous-doc rollback on insert
  failure (index/cache drift); native `type` filtering in task-only
  `search()` (was JS post-filter, contradicting ADR 0079); embedding-config
  validation on index load (BM25-only cache + hybrid caller → rebuild);
  snippet generation now uses the compound tokenizer (a "FeatureStore" hit
  is no longer invisible to its own snippet).
- **Golden-suite regression guards** (`search-golden.test.ts` §ADR-0083):
  rank-normalization survival, exact-ID rank-1 across query shapes,
  title-pin behavior, conservative single-token pin. Plus +11 scoring unit
  tests (tie semantics, anti-annihilation, pin ordering).

### Phase B — memory as substrate

- **`memory` substrate** (`shared/src/substrates/memory.ts`, prefix `MEMO`):
  `layer` (episodic | semantic | procedural), `source`, `entity_refs`,
  `tags`, `valid_until`, `usage_count`, `supersedes`. Body = the memory
  content. No `status` — validity is `valid_until`, per the "never overload
  status" principle (ADR 0097). One declaration; ID pattern, validation,
  viewer rendering, and search indexing all derived (ADR 0098's promise,
  validated in practice).
- **`BacklogMemoryStore`** (`server/src/memory/backlog-memory-store.ts`) —
  the default `MemoryStore` behind the unchanged ADR 0092 plugin boundary:
  - `store` mints `MEMO-` entities through the service (lazy service
    resolution so composer wiring doesn't force singleton construction).
  - `recall` rides `searchUnified` restricted to `type: memory`, then
    applies layer/context/tags/expiry filters; over-fetches 3× so the
    filters don't starve results.
  - `forget` is **soft by default** (`valid_until = now` — recall drops it,
    the viewer keeps it auditable); `{ expired: true }` hard-deletes (the
    GC path). OR-semantics mirror `InMemoryStore.forget` for drop-in parity.
  - The transient `session` layer is rejected — session memory dies with
    the process *by design*.
- **Wiring**: bootstrap registers one store instance for all three persisted
  layers (composer dedupes instances); capture hooks unchanged (ADR 0092.2
  contract intact); `core/recall` defaults to all persisted layers;
  memories excluded from generic `backlog_search`/`backlog_list`/task
  search unless explicitly requested — `backlog_recall` is the read
  surface; `entity_refs` + `tags` indexed into searchable text so
  recall-by-referenced-id works; "memory"/"memories" added to query-intent
  type words; viewer renders MEMO entities.
- **R1–R5 contract suite** (`memory-store-contract.test.ts`, 11 tests)
  running against the *real* Orama pipeline, not stubs: durability across
  store instances, production-pipeline ranking, exact-match by MEMO- id and
  referenced entity id, filter composition, expiry semantics, soft/hard
  forget, and a capture→entity→core-recall E2E.

## Distilled insights (worth more than the code)

- **The flagship feature was structurally dead, not flaky.** CLI `recall`
  could never return results: capture and recall ran in different processes
  with per-process `InMemoryStore` heaps. No amount of tuning would have
  fixed it — the architecture was the bug. Durability had to come first.
- **Two "strengths" tests were powered by an accident.** The golden tests
  "finds task by full ID" and "searches epic_id field" passed only because of
  the id-field token noise that ADR 0083 #4 condemned as a design issue. The
  same accident also made bare-number queries "work". Removing the accident
  required rewriting the tests to assert the *intended* contract (ID queries
  are navigational; children are a filter query) — golden tests encode
  behavior, including behavior you didn't mean to have.
- **One pre-existing golden failure was environment-masked.** "search →
  EPIC-0002 above TASK-0005" fails on pristine HEAD in BM25-only mode; it
  passes only when the vector retriever contributes. BM25 cannot express
  "aboutness" (ADR 0083 Part 3) — the assertion was rewritten
  retriever-agnostic, and strict aboutness ranking stays deferred to the
  cross-encoder.
- **The substrate architecture paid for itself.** Adding `memory` touched
  exactly the files ADR 0098 promised: one substrate module + enum key +
  registry entries. Storage, ID utilities, validation, query-intent
  prefixes, and viewer metadata all derived. The only manual viewer change
  was an icon.
- **Adapter over engine.** `BacklogMemoryStore` is ~200 lines and inherits
  durability, hybrid ranking, decay, snippets, SSE, operation logging, and
  D1 parity. The rejected `OramaEpisodicStore` sidecar would have
  re-implemented or re-wired each of those (and doubled the ADR 0101
  reconciliation problem).
- **Prefix correction**: ADR 0092.3 sketched `MEM-`; the codebase convention
  is uniformly 4-char prefixes (TASK, FLDR, MLST, CRON) and ID tooling
  assumes it — shipped as **`MEMO-`**.
- **TS 5.5+ inferred type predicates bite generic layer code**:
  `.filter(l => l !== 'session')` silently narrows the element type,
  breaking `.includes(layer)` on the wider union. Explicit annotation at the
  declaration site is the fix.

## Recall-correctness contract status (R1–R5)

| # | Property | Status |
|---|---|---|
| R1 | Durable across processes/transports | ✅ memories are markdown entities; contract test covers fresh-instance recall |
| R2 | Ranked by the production hybrid pipeline | ✅ recall rides `searchUnified`; rank-fusion + decay + pin |
| R3 | Exact: MEMO- id, referenced entity id, distinctive phrase → rank 1 | ✅ intent short-circuit + entity_refs indexing + title pin |
| R4 | Scoped: context/layers/tags filter | ✅ JS-filtered on entities (native Orama fields deferred — corpus is small) |
| R5 | Time-honest: expired excluded; recency without exactness loss | ✅ `valid_until` filtering; decay tested with pin guard |

## Potential next phases (from ADR 0092.3, sharpened by this work)

- **Phase C — explicit memory verbs + wakeup unification.**
  `backlog_remember` (MCP + CLI; the write-side for semantic/procedural
  knowledge, with `supersedes` correction lineage) and `backlog_forget`
  (exposing the now-working soft forget). Wakeup gains the knowledge
  section (top semantic/procedural memories for the scope, ≤150 of the
  ~600-token budget). After C, the Part-1 session loop of 0092.3 is
  executable end-to-end. Small: both tools are thin wrappers over
  now-existing core capability.
- **Phase D — consolidation (episodic → semantic/procedural).**
  Server ships `backlog_consolidation_candidates` (clusters of related,
  aged episodics); a `cron` entity (ADR 0097 Extension 1) describes the
  recurring job; an external agent distills via `remember` and retires via
  `forget`. The LLM lives in the agent, not the server — ADR 0092's
  no-LLM-write-path constraint holds.
- **Phase E — feedback loop on durable ground.** Echo/fizzle flushing to
  `usage_count` frontmatter; `applyUsageSignal` next to decay in scoring;
  GC policy (consolidated + fizzled + expired → forgotten). Open design
  question flagged in 0092.3: recall returns full content, so the original
  follow-up-call echo signal (0092.1 §Q4) under-counts — the echo design
  needs a revisit before building.
- **Deferred, explicitly**: cross-encoder re-ranking (ADR 0083 #9 — the
  remaining "aboutness" ceiling; ~22MB model, +100-200ms, behind a flag);
  native Orama fields for layer/tags if memory corpora outgrow JS
  filtering; `include_expired` recall param; D1 worker composer wiring
  (the substrate makes it possible; it is not yet wired).
