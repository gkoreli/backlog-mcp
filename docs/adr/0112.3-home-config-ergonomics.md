# Fix Home Config Ergonomics

Sibling to the dir-layout proposal (which handles `.backlog-mcp`‚Üí`.backlog` and
the global nested control dir). This one is about **config files and env vars**,
not directories. Two fixes, both settled before Phase E.

**1. Global config is one flat `config.json` ‚Äî no `config.local.json` split.**

The two-file split (`config.ts:15-16`, merged in `loadRepoConfig` at
`config.ts:97-107`) is a **collaboration** feature, exactly like VS Code's
`.vscode/settings.json` + personal override:

- `config.json` ‚Äî committed, the team's shared default.
- `config.local.json` ‚Äî gitignored, your machine-only override.

The split only earns its keep where collaboration happens ‚Äî a project in git.
The global home (`~/.backlog`) is a private drawer: no repo, no team, nothing
committed. There, both files mean the same thing ("my settings"), so the second
is pure confusion. Global loads a single flat `config.json`; the base+override
merge applies to project homes only.

**2. Align `BACKLOG_SCOPE` ‚Üí `BACKLOG_CONTEXT` (and the config `scope` key).**

`BACKLOG_SCOPE` (`config.ts:17`, resolved at `config.ts:128-143`) sets the
default **sub-folder** inside a home ‚Äî a container id like `FLDR-0001` that
pre-filters `wakeup`/`recall`/`remember`. It is *not* home selection.

ADR 0112 R-8 explicitly renamed this dimension "context" to stop it being
overloaded with "home" (0112:379-385). But the env var and config key still use
the old ADR 0105 word `scope`, sitting next to the new `BACKLOG_HOME` /
`BACKLOG_PROJECT_ROOT` ‚Äî three env vars, two vocabularies, one confusion. Rename
the env var to `BACKLOG_CONTEXT` and the config key `scope`‚Üí`context` to match
the shipped API vocabulary. Phase E owns the migration since it's already moving
config.

## Why cheap

Both are constructor/resolution-layer changes, not new code paths:

- Global-flat-config: skip the `config.local.json` overlay when the home is
  global ‚Äî one branch in the loader.
- scope‚Üícontext: a rename with a Phase E migration; the resolution precedence in
  `resolveScope` (`config.ts:128-143`) is unchanged, only the key/var names.

Underlying principle (shared with the layout proposal): the committed-vs-personal
machinery is a project-only feature; global doesn't need it. And user-facing
names should match the shipped vocabulary, not the superseded one.
