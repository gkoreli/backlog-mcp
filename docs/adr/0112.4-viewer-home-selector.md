# Viewer Home Selector + Cross-Home Search

Give the viewer a way to show and switch which backlog *home* it's looking at.
The data plumbing already exists — `HomeSelection`, `getHomeId`,
`getHomeSelection`, `HomeProvenance` in `packages/viewer/utils/api.ts`, and every
downstream component (`task-list`, `task-detail`, `document-view`,
`activity-panel`, `metadata-card`) already renders per-home. The only missing
piece is the human-facing control. Build that, and nothing else.

## Depends on Phase E

This is only *useful* once the default server resolves project homes from the
request (Phase E cutover — see ADR 0112). Today the published daemon still serves
the global singleton, so a "project" selection resolves to the same global store.
Sequence: land Phase E, then this. Treat them as one unit or land E first.

## Non-negotiable: the daemon stays home-agnostic

One daemon, one port, stateless per request (`hono-app.ts:102-108` reads home
from `X-Backlog-Home`/`X-Backlog-Project-Root` headers or `?home`/`?project_root`
query per request). The selector must **never** ask the server "what workspaces
exist?" and must never trigger a filesystem scan (ADR 0112 R-2/R-9). It offers
only homes it legitimately knows: global (always) + whatever the URL/session
already carries.

## 1. Provenance badge (primary role, always visible)

A badge in the viewer chrome (next to the theme toggle) reading the
`HomeProvenance` (`home`, `home_id`, `source_path`) the server already returns:

```
🌐 global            when home=global
📁 backlog-mcp       when home=project (dir name from source_path)
```

Its first job is not switching — it's making sure a human never mistakes a global
task for project work (R-9). This alone is worth shipping even before switching.

## 2. Switch = rewrite the URL (secondary role)

The active home lives in the viewer's **URL** — the app is a pure function of it.
Clicking the badge opens a small menu:

1. **Global** — always. `→ /?home=global`
2. **This project** — only if the viewer's URL carries a `project_root` (i.e. the
   bridge/CLI opened it from a repo). `→ /?home=project&project_root=<path>`
3. **(later) Recently opened** — homes this daemon was *explicitly pointed at*
   this session, from the runtime registry. Never a disk scan.

Switching navigates (rewrites the URL); it does not manage state.
`getHomeSelection()` already parses the URL, every component already reacts, and
`getHomeId()` already gives each home a stable cache key. New surface = one chrome
component (`home-selector.ts`) over existing types. No new endpoint for v1.

## 3. Cross-home lives in search, NOT the browse view

Do **not** merge homes into one default list — that rebuilds the ~1,000-entity
global pile the whole docs-native pivot exists to escape, and violates "artifacts
belong to the project" (Invariant 6). The default view is **one home at a time.**

The merge belongs only in discovery, and the capability already exists: `home:
all` (R-9) is read-only, rank-fused via the shipped `cross-home-rrf` merger, and
provenance-stamped. Wire it into spotlight/search as an opt-in "search across all
homes" toggle:

- results render with a home badge on every row;
- clicking a result **navigates into that home** (rewrites the URL) rather than
  editing it inside the blend;
- writes are never cross-home.

Separate to browse; merged to find.

## Scope

- New: `packages/viewer/components/home-selector.ts` (badge + URL-switch menu).
- Change: spotlight/search gains a cross-home toggle rendering provenance-badged
  results that navigate on click.
- No server change for v1 (global + current-project). A `/homes` endpoint listing
  session-known homes is a *later* enhancement, and only ever session-scoped —
  never a machine crawl.
