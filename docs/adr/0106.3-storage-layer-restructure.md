# 0106.3. Storage Layer Restructure — Collapse the Dead Adapter, Rename `TaskStorage`→`FilesystemStorage`, Split `local/` vs `d1/`

**Date**: 2026-06-17
**Status**: Accepted (execution delegated)
**Backlog Item**: TASK-0688
**Thread**: applies ADR 0106.1 glossary + 0106.2 rename; establishes file-naming convention (also in AGENTS.md)

## Problem — three issues in `packages/server/src/storage/`

### 1. The local adapter is dead code; the local service bypasses the port
- `FilesystemStorageAdapter` (`filesystem-adapter.ts`) `implements StorageAdapter`
  and is a **pure pass-through** wrapper over `TaskStorage` — every method is
  `return this.storage.x()`.
- **Nothing imports it.** Grep for `FilesystemStorageAdapter` outside its own file
  and the interface = **zero hits**.
- Meanwhile `BacklogService` (the real local service) does `private taskStorage =
  new TaskStorage()` (`backlog-service.ts:17`) — it depends on the **concrete
  class**, not the `StorageAdapter` interface.
- Asymmetry: the cloud path is clean (`D1BacklogService` → `D1StorageAdapter
  implements AsyncStorageAdapter`), the local path is not. A half-finished
  port/adapter refactor: the adapter abstraction was built, then never adopted.

### 2. `TaskStorage` is mislabeled (Task-as-Entity rot, per ADR 0106.1/0106.2)
- It stores **all entity types** (returns/accepts `Entity`), but is named
  `TaskStorage` with `TASKS_DIR='tasks'`, `taskFilePath`, `taskToMarkdown`,
  `markdownToTask`, `iterateTasks`, `task:` params, `allTasks`. Fossils from when
  tasks were the only type.

### 3. Flat folder mixes two adapter families + inconsistent file naming
- `local` (filesystem) and `d1` (cloud) files sit flat in `storage/`.
- File naming is inconsistent: `service-types.ts` (dash) vs the repo's dominant
  `types.ts`. `service-types.ts` is in fact the **`IBacklogService` contract**,
  implemented by both services and imported by ~40 files — its name understates
  what it is.

## Decision

### A. Collapse the dead wrapper (maintainer-approved: "Collapse")
- **Delete** `filesystem-adapter.ts` (the do-nothing wrapper — currently the class
  named `FilesystemStorageAdapter`).
- **Rename** `TaskStorage` → **`FilesystemStorage`** and have it **directly
  `implements StorageAdapter`**. One class, one job; removes a layer that earned
  nothing (SOLID: every class justifies its existence).
- **Naming (maintainer choice)**: the concrete class is `FilesystemStorage`, NOT
  `FilesystemStorageAdapter`. The pattern term ("Adapter") lives on the *interface*
  (`StorageAdapter`); a concrete class naming the pattern instead of the thing is
  noise. `implements StorageAdapter` already expresses the adapter role. For
  symmetry, **`D1StorageAdapter` → `D1Storage`** too (interfaces keep `*Adapter`).
- **`BacklogService` depends on the `StorageAdapter` interface**, not the concrete
  class — `private storage: StorageAdapter = new FilesystemStorage()`.
  Restores dependency inversion and makes local symmetric with the D1 path.

Rejected alternative — "keep both, wire the wrapper in": preserves a pass-through
layer that adds zero behavior. Activating dead code without justifying it is worse
than deleting it.

### B. Rename Task-as-Entity fossils in the storage impl (continues ADR 0106.2)
- `TASKS_DIR='tasks'` → keep the on-disk dir name **`tasks`** for backward
  compatibility (data already lives there — do NOT migrate files), but rename the
  **code** symbols: `taskFilePath`→`entityFilePath`, `taskToMarkdown`→
  `entityToMarkdown`, `markdownToTask`→`markdownToEntity`, `iterateTasks`→
  `iterateEntities`, `task:` params → `entity:`.
- **Gotcha (must preserve)**: the on-disk directory is literally `tasks/`. Renaming
  the *constant's value* would orphan every existing file. Rename the symbol, keep
  the string `'tasks'`. Document this inline.

### C. Subfolder split
```
storage/
├── storage-adapter.ts             → StorageAdapter, AsyncStorageAdapter, ListFilter (shared interfaces)
├── backlog-service.contract.ts    → (was service-types.ts) IBacklogService contract
├── entity-factory.ts              → (was schema.ts) createEntity (ADR 0106.2)
├── local/
│   ├── filesystem-storage.ts           → (was task-storage.ts) class FilesystemStorage implements StorageAdapter
│   └── backlog-service.ts              → BacklogService, depends on StorageAdapter
└── d1/
    ├── d1-storage.ts                   → (was d1-adapter.ts) class D1Storage implements AsyncStorageAdapter
    └── d1-backlog-service.ts
```

### D. File naming convention (also added to AGENTS.md)
- **Satellite types** (serve one sibling) → `<base>.types.ts`, co-located.
- **Module-wide types** → folder `types.ts` (keep existing `core/`, `context/`,
  `resources/`, `operations/`).
- **Shared contracts** (implemented by many, consumed widely) → `<name>.contract.ts`.
  → `service-types.ts` becomes **`backlog-service.contract.ts`** (maintainer choice).
- No sweeping repo-wide rename — apply to touched files only; migrate legacy
  opportunistically (churn + merge-conflict risk otherwise).

## Interface cleanup (do while touching)
- `StorageAdapter`/`AsyncStorageAdapter`/`IBacklogService` use `add(task: Entity)`,
  `iterateTasks()`, and `counts()` returns `total_tasks`/`total_epics`. The method
  params (`task`→`entity`) and `iterateTasks`→`iterateEntities` should be renamed.
- **`counts()` keys `total_tasks`/`total_epics`**: these may be consumed by the
  viewer/API as a wire contract. **Do NOT rename these keys without checking
  consumers** (viewer, `mcp__backlog__backlog_list` counts). If consumed, leave the
  keys; only rename internal locals. Flag any change in the audit.

## Non-goals
- No data migration (on-disk `tasks/` dir stays).
- No behavior change, no MCP tool I/O change.
- Not the semantic intent-verbs (ADR 0106).

## Brutal critique / risks
1. **`tasks/` dir constant** — renaming its value orphans data. Mitigation: rename
   symbol, keep value `'tasks'`; covered above.
2. **`counts()` wire keys** — `total_tasks`/`total_epics` may be a public contract.
   Mitigation: audit consumers before touching; default to leaving keys.
3. **Import churn** — `backlog-service.contract.ts` rename touches ~40 importers;
   subfolder moves touch every storage import. Mitigation: mechanical, typecheck-
   gated; do moves and renames in separate commits for legible history.
4. **Overlap with ADR 0106.2** — both edit `backlog-service.ts`, `task-storage.ts`,
   `schema.ts`. Mitigation: **0106.2 and 0106.3 are MERGED into one delegated
   execution** (TASK-0688) to avoid collision — see Sequencing.
5. **Singleton wiring** — `BacklogService` is a singleton with a private
   constructor; injecting the adapter must preserve singleton semantics and the
   search/cache bootstrap. Mitigation: keep construction internal; only change the
   field's *type* to the interface + concrete instantiation.

## Sequencing — merge with ADR 0106.2
0106.2 (vocabulary rename) and 0106.3 (storage restructure) touch the same files and
0106.3 *renames the files 0106.2 edits*. Running them separately = guaranteed conflict.
**Decision: one delegated change** covering both, in this order:
1. ADR 0106.2 §3 factory cleanup: `schema.ts`→`entity-factory.ts`, `createTask`→
   `createEntity`, derive input from `z.input<EntitySchema>`, drop if-ladder.
2. `<Type>Entity` aliases in `shared` (ADR 0106.2 §2).
3. Storage collapse + rename (this ADR §A,§B).
4. Subfolder split (§C) + contract rename (§D).
5. Remaining Task-as-Entity renames across core/tests (ADR 0106.2 §1).
Run the full suite between chunks.
