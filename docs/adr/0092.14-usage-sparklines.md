# 0092.14. Usage Sparklines

**Status**: Accepted ΓÇõ shipped 2026-06-16 (`92ea7cd`). Verified: 5/5
core-usage-series tests green (touch detection across id/ids, day
bucketing oldestΓÁÈnewest, window + malformed-line skip, empty series,
hasUsage). Server + viewer typecheck clean; viewer 96 tests +
viewer-routes green. Sandbox search/orama + `sharp` suite failures are
the macOSΓÁÈLinux native-binary mismatch, not regressions. Live-viewer
visual spot-check of the sparkline: recommended next server run.

**As-built note**: the sparkline shares the contradiction chip's surface
ΓÇõ the REST detail pane (`/tasks/:id` ΓÁÈ `DocumentView` ΓÁÈ `MetadataCard`)
ΓÇõ since `usage_series` is computed server-side, not stored `.md`
frontmatter. The same two-panes-chip-rendering follow-up logged in
0092.13 covers unifying this with the resource-viewer pane.

**Why now**: ADR 0092.9 made memory usage *recorded* ΓÇõ every recall,
expand, and cite appends to `memory-usage.jsonl` (R-16/R-14), and strong
events bump a relatime-gated `usage_count` (R-13). ADR 0092.11 surfaced
evidence chains and validity but deferred the usage view. This loop makes
the recorded history *visible*: a per-memory sparkline of daily touches.
It is the smallest remaining piece of the human-visibility half of the
north star ΓÇõ the JSONL we already write, finally rendered.

**Evidence basis** (0092.9 R-13/R-14): usefulness is composite ΓÇõ strong
expand/cite signals plus weak recall touches. The durable `usage_count`
is deliberately lossy (Fibonacci-bucketed flushes); *"the JSONL holds the
exact history"*. So the truthful usage view reads the JSONL, not the
frontmatter counter ΓÇõ the counter says *how much*, the sparkline says
*when*.

## Design

- **Pure fold**: `usageSeries(lines, id, {windowDays: 30, now})` ΓÁÈ
  `number[]` of per-day counts of events that *touched* `id` (scalar
  `id`, or membership in `ids[]` ΓÇõ recall, expand, cite all count).
  Oldest bucket first, newest last (leftΓÁÈright). Deterministic, malformed
  lines skipped. `hasUsage(series)` lets callers omit empty sparklines.
- **IO at the edge** (ADR 0090, same seam as 0092.12's demand reader):
  `AppDeps.readUsageLines?` is injected by the Node entry from
  `bootstrap.readUsageLines`; the Worker omits it ΓÁÈ no sparkline, no
  Node-fs in the bundle. The memory detail (`/tasks/:id`) attaches
  `usage_series` only when present and non-empty.
- **Generic viewer rendering** (the 0092.11 lever, again): `MetadataCard`
  gains *one* rule ΓÇõ **an array of all-numbers renders as an inline SVG
  bar sparkline**. Key-agnostic, so it needs no coupling to the field
  name (which `DocumentView`'s `_`ΓÁÈspace transform would mangle anyway).
  Bars scale to the series max; a total label reads "how used" at a
  glance. Sits naturally beside the existing `usage_count` field.

No new endpoint, no client fetch path, no charting dependency ΓÇõ a handful
of `<rect>`s. The store still doesn't act (ADR 0097): the series is
recomputed on read from the append-only log.

## Deferred (next loops)

- **Split-by-type sparkline** (recall vs. expand vs. cite as stacked or
  colored bands) ΓÇõ the data supports it; one color ships first.
- **A usage column in the memory list** (sparkline per row) once the
  detail view proves the read pattern.
