# 0092.2. Agentic Memory Phase 3 — Implicit Episodic Capture

**Status**: Proposed
**Target ADR**: 0092 (plugin-based agentic memory architecture) — Phase 3
**Supersedes**: ADR 0092.1 §"Phase 3 — Implicit Memory Capture" (that plan was
a best-guess skeleton; this document locks the real contract after auditing
the current code.)
**Defers to a future ADR**: LLM-based memory re-extraction, cross-session
echo tracking (those are Phase 4+ concerns).

## Why this ADR exists

ADR 0092 said *what* Phase 3 should do ("hook `update → done` and
`create → artifact`, write memories"). ADR 0092.1 §Phase 3 sketched *where*
those hooks go. But neither answered the questions that actually matter:

1. What does an **episodic memory** do that the backlog doesn't already do?
   `backlog_wakeup` (Phase 2) already surfaces recent completions by reading
   tasks directly. If the backlog IS the source of truth, why write a second
   record?

2. The current `MemoryStore`/`MemoryComposer` are **untested, unwired, and
   unused by any production code path**. Phase 3 is not "add a line to
   `update()`" — it's the first real production integration of the memory
   subsystem. The write boundary we pick here shapes Phase 4 and 5.

3. `TASK-0676` (search ranking bug) surfaced during the Phase 2 triage
   session — the current search pipeline cannot reliably find ARTF-0174 even
   by exact ID. If search is broken, a "pointer-only" memory (as 0092.1
   proposed) is a broken pointer. We need to decide whether Phase 3 proceeds
   on top of known-broken search, or whether `TASK-0676` blocks Phase 3.

This ADR answers all three before any code gets written.

## Architecture audit — where we actually stand (2026-05-07)

Before deciding, a ground-truth read of every component Phase 3 touches.

### The memory package (`packages/memory`)

| Component                | Path                                           | Status                                                                |
|--------------------------|------------------------------------------------|-----------------------------------------------------------------------|
| `MemoryEntry` + layers   | `src/types.ts`                                 | Defined. `layer: 'session' \| 'episodic' \| 'semantic' \| 'procedural'`.         |
| `MemoryStore` interface  | `src/types.ts`                                 | Defined: `store`, `recall`, `forget`, `size`.                         |
| `MemoryComposer`         | `src/composer.ts`                              | Works — routes by layer, merges recall results by score.              |
| `InMemoryStore`          | `src/in-memory-store.ts`                       | Zero-dep reference impl. Keyword scoring, in-process storage only.    |
| `MemPalaceStore`         | `src/mempalace-store.ts`                       | Optional pythonia bridge. Not our path — kept as a plugin option.     |
| **Tests**                | —                                              | **ZERO**. No `*.test.ts` files in `packages/memory/src/`.             |
| **Production usage**     | —                                              | **ZERO**. No server code imports `MemoryComposer` or `MemoryStore`.   |

The memory package ships dead code today — interfaces + a reference store that
nobody uses. Every use of `@backlog-mcp/memory` from the server is for the
**search** sub-package (`OramaSearchService`, `scoring`). Memory itself is
unreferenced.

**Implication:** Phase 3 is not "hook into existing flow." It's "stand up the
first flow." That changes the size estimate in ADR 0092.1 (~100 LoC → realistic
~300 LoC including wiring + tests) and it changes the risk posture.

### The backlog service (`packages/server/src/storage/backlog-service.ts`)

- Singleton class with no DI for cross-cutting concerns today.
- `add()` and `save()` accept an `Entity` and update the search index. No
  actor, no context, no memory composer.
- The write boundary (`create`, `update`, `delete`) lives in **core**
  (`packages/server/src/core/*.ts`), not in the service. Service is the
  persistence layer; core is the orchestrator.

### The core write path (`packages/server/src/core/`)

`createItem` and `updateItem` already take a `WriteContext` with `actor`,
`operationLog`, `eventBus`. The transport layers (MCP tool handlers, CLI
commands) build the context per request and pass it down. This is exactly
the seam Phase 3 needs — memory capture is a post-write side effect that
needs `actor` attribution and must fire iff the mutation succeeded.

```ts
// packages/server/src/core/types.ts — the existing write boundary
export interface WriteContext {
  actor: Actor;
  operationLog: IOperationLog;
  eventBus?: { emit: (...) => void };
}
```

**Implication:** Phase 3 extends `WriteContext`, not `BacklogService`. The
composer is one more optional dependency carried by the context — the same
transport-agnostic pattern that ADR 0094 established for the operation log.
This was missed in ADR 0092.1 (which proposed hooking inside the service).

### The session-memory stage (`packages/server/src/context/stages/session-memory.ts`)

Ironically, this file is *called* "session-memory" but has nothing to do with
the `packages/memory` subsystem. It derives a session summary from the
operation log for `backlog_context` payloads. No memory writes, no recall.

**Implication:** Naming collision is survivable but worth documenting. The
"session" layer in `MemoryLayer` is different from the "session" concept in
this file. Phase 3 writes **episodic** memories; this file's "session" is an
operation-log-derived ephemeral concept.

### Search (`packages/memory/src/search/orama-search-service.ts`)

- BM25 + vector hybrid via Orama.
- `linearFusion → applyTemporalDecay → applyCoordinationBonus` pipeline
  (Phase 1 shipped).
- **Known broken:** `TASK-0676` — exact-ID lookups, title substring matches,
  and distinctive body phrases all fail to surface the correct entity.
  Reproduced live during the Phase 2 triage session.

**Implication:** If memories are pointers and search is broken, recall is
broken. This ADR has to decide whether Phase 3 ships on top of known-broken
search or blocks on `TASK-0676`.

## The vision re-grounded

ADR 0092's thesis was: **the backlog IS the memory; what's missing is memory
*behavior*.** Phase 3 is the first phase that writes *new* records — everything
so far (Phase 1 decay, Phase 2 wakeup) was pure read-side behavior on
existing data. Before writing even one episodic entry, we owe it to the
architecture to answer: **what does this new record do that reading the
backlog directly doesn't?**

### Three candidate answers

**Answer A — Indexing shortcut.**
"We index evidence into a separate Orama collection optimized for short-text
semantic recall, so memory recall is faster and more relevant than full-task
search." *Requires:* second Orama collection, separate schema, duplicate
embeddings.

**Answer B — Curated slice.**
"The backlog contains all tasks; memories are a *curated* subset — only
completions with real evidence, only artifacts, etc. — that's worth surfacing
without the noise of every open ticket." *Requires:* selective capture
logic, but no new storage (the in-memory store is enough for this).

**Answer C — Layer composition.**
"Memories exist to be *recomposed* with session/procedural/semantic layers
via `MemoryComposer`. The value is the composer's cross-layer merge, not any
single layer's contents." *Requires:* at least two layers populated before
the abstraction earns its complexity.

**This ADR commits to Answer B.**

Rationale: A is premature optimization — full-task search already works
well for 600-task corpora (except for `TASK-0676`'s specific bugs). C is
aspirational — we have no session, semantic, or procedural stores today, so
composition has nothing to compose. B is the minimum-viable justification:
capture events give us a *filtered* view of the backlog that's useful on
its own, and Answer C becomes available later without rewriting Phase 3.

### What "episodic memory" means in backlog-mcp after this ADR

An **episodic memory** is a **pointer + digest** record written at the moment a
significant event happens in the backlog. Specifically:

- **Pointer**: `metadata.entity_id = "TASK-0629"` — the canonical reference
  to the backlog entity that triggered the event.
- **Digest**: `content = "<title> — <first evidence line>"` (~200 chars) —
  a human-meaningful summary that can be scanned without resolving the
  full entity. Answers "what was this?" without a second lookup.
- **Context**: `context = parent_id` — enables scoped recall (e.g., "what
  did I complete inside `FLDR-0001`?").
- **Tags**: `[entity.type, ...entity.tags]` — carries the substrate and
  user-added tags for filterable recall.

The digest is **not a lossy summary** of the evidence — it's the first line
verbatim. No LLM. No re-extraction. The agent already wrote structured
evidence; we index the leading line as the short form.

### What triggers capture

Two events, both in the core write boundary (not the service):

| Event                  | Source                     | When                                              |
|------------------------|----------------------------|---------------------------------------------------|
| Task completion        | `core/update.ts`           | `status` transitions to `done` (and was not done) |
| Artifact creation      | `core/create.ts`           | `type === 'artifact'`                             |

These match the original 0092.1 selection. The rationale for *both*:

- **Completions** mark a real event ("this got finished"). They carry evidence
  — the richest agent-produced content in the backlog.
- **Artifacts** are explicitly memory-shaped entities — the user or agent
  already decided "this deserves to be kept." Not capturing them would be
  deliberately forgetting.

Tasks that go `in_progress` / `blocked` / `open` are *not* captured. Those
states come and go; they're not events worth memorializing.

### What doesn't trigger capture (and why)

| Non-trigger                    | Why not                                                                     |
|--------------------------------|------------------------------------------------------------------------------|
| Every `update` call            | Most updates are noise (status wobbles, typo fixes, parent reshuffles).     |
| Task creation (type = task)    | A task is a promise, not an event. Capture is for things that *happened*.    |
| Task deletion                  | A deletion is a revision of history, not a memory to accumulate.            |
| Every `write_resource`         | Resources are often structured docs; if they're artifacts, the artifact hook captures them.  |

## Decisions

### D1 — Extend `WriteContext` with `memoryComposer?: MemoryComposer`

Add the composer to the per-write context, not to `BacklogService`:

```ts
// packages/server/src/core/types.ts
export interface WriteContext {
  actor: Actor;
  operationLog: IOperationLog;
  eventBus?: { emit: (event: ...) => void };
  memoryComposer?: MemoryComposer;  // new — optional, see D3
}
```

**Why context, not service:** memory capture is attribution-aware (who
completed this?) and transport-varied (CLI session vs. MCP session vs. cron
tick each have different composers in a hypothetical future). The service
is the persistence layer — making it hold a composer forces it to know
about actors and sessions, which ADR 0094 already decided it shouldn't.

**Trade-off:** every transport adapter now has one more field to populate.
Acceptable — they already build the context with `actor` and `operationLog`.

### D2 — Capture happens inside core, not the service, not a wrapper

```ts
// packages/server/src/core/update.ts — extension
await service.save(validated);

if (ctx.memoryComposer && shouldCapture(prev, validated)) {
  await captureCompletion(ctx.memoryComposer, validated, ctx.actor);
}

recordMutation(ctx, 'backlog_update', ...);  // existing
return result;
```

**Why inside core:** capture is a consequence of a specific business-logic
transition (`→ done`). The rule "did status change?" requires comparing
`prev` and `validated` — only core has both. Putting capture in the service
forces it to diff entities, which it doesn't do today.

**Why not a wrapper:** a wrapper (e.g., `withMemoryCapture(updateItem)`) is
a middleware pattern — it adds a layer of indirection that can silently fail
("write succeeded, memory capture errored, caller doesn't know"). Capture
is part of the semantics of "complete a task"; inlining it makes that
visible.

**Order matters:** capture fires *after* `service.save()` succeeds and
*before* `recordMutation()`. Rationale: a capture is conceptually part of
the operation, so it should precede the journal entry (if capture errors,
we don't want to journal an inconsistent state). The operation log is the
audit record; the memory is a derived side effect.

**Fire-and-forget?** No. Capture is awaited. A failed capture should not
corrupt the response, but it should surface in logs. See §D7 for error
handling.

### D3 — Composer is optional

`ctx.memoryComposer` is nullable. Transports that don't care about memory
(low-throughput scripts, tests, edge cases) can omit it. Core checks for
presence:

```ts
if (ctx.memoryComposer) { ...capture... }
```

**Why optional:** it preserves backward compatibility for every existing
test and every transport that hasn't been updated. The viewer, the D1
worker, any cron jobs — none need to touch memory on day one.

**Default production wiring:**
- Node server: wires a real `MemoryComposer` with `InMemoryStore` registered
  for `layer: 'episodic'`.
- CLI: same default.
- D1 worker: omits the composer in Phase 3 (memory is Node-only initially).
- Tests: pass a stub or omit entirely.

### D4 — `InMemoryStore` is fine for Phase 3. Don't build `OramaEpisodicStore` yet.

Phase 3 ships with `composer.register('episodic', new InMemoryStore())`.

**Why InMemoryStore first:**
1. **Correctness before complexity.** Get the capture semantics right (the
   right events, the right record shape, the right error handling) before
   picking a persistent backend.
2. **Process-bounded state is enough to validate the UX.** A session of
   `backlog-mcp wakeup` followed by `backlog_search --memory-only` (Phase 4)
   can demonstrate value without durability.
3. **Orama-backed store is a drop-in replacement.** The `MemoryStore`
   interface is the contract; swapping stores doesn't change any capture
   code. Phase 3.1 (or as part of Phase 4) can add `OramaEpisodicStore`.
4. **`TASK-0676` blocks a high-quality Orama-backed memory store anyway.**
   If Orama full-text search is broken for exact IDs and title substrings,
   it will be broken for memory recall too. Build on a known-good path
   first.

**What we lose:** memories don't survive a server restart. Acceptable for a
single-user tool; capture is cheap and the early corpus is small.

### D5 — Memory entry shape (final)

```ts
// Completion memory (triggered by update → done)
{
  id: `mem-${entity.id}-${now}`,
  layer: 'episodic',
  content: `${entity.title} — ${entity.evidence?.[0] ?? ''}`.slice(0, 200),
  source: actor.name,
  context: entity.parent_id,            // scope for recall filtering
  tags: [entity.type, ...(entity.tags ?? [])],
  createdAt: now,
  metadata: {
    entity_id: entity.id,
    kind: 'completion',
    actor_type: actor.type,             // 'user' | 'agent' — lets recall weight accordingly
  },
}

// Artifact-creation memory (triggered by create when type === 'artifact')
{
  id: `mem-${entity.id}-${now}`,
  layer: 'episodic',
  content: `${entity.title} — ${(entity.description ?? '').slice(0, 160)}`,
  source: actor.name,
  context: entity.parent_id,
  tags: ['artifact', ...(entity.tags ?? [])],
  createdAt: now,
  metadata: {
    entity_id: entity.id,
    kind: 'artifact',
    actor_type: actor.type,
  },
}
```

Changes vs ADR 0092.1:
- Added `metadata.actor_type` — Phase 4 may weight agent-authored vs.
  user-authored memories differently; cheaper to capture now than retrofit.
- Slightly shortened artifact digest (160 instead of title+200 — artifact
  titles can be long; leaves headroom for description excerpt).

### D6 — `shouldCapture()` is explicit, testable

```ts
// packages/server/src/memory/capture-rules.ts
export function shouldCaptureCompletion(prev: Entity, next: Entity): boolean {
  return next.status === 'done' && prev.status !== 'done';
}

export function shouldCaptureArtifact(entity: Entity): boolean {
  return entity.type === 'artifact';
}
```

Dedicated file so the rules have a single home. A unit test table validates
every transition: `open → done`, `in_progress → done`, `done → done` (no
capture), `done → open` (no capture), etc.

### D7 — Capture errors don't fail the write

```ts
if (ctx.memoryComposer) {
  try {
    await captureCompletion(...);
  } catch (err) {
    console.error('[memory] capture failed', { id: entity.id, err });
    // no re-throw — the mutation itself succeeded
  }
}
```

**Why not fail the write:** the user's action (complete a task) succeeded.
Failing the tool call because a derived side-effect broke would confuse
every caller. The mutation is durable in the backlog; the memory is a
bonus.

**What about silent data loss?** Acceptable at this phase. Phase 4 (echo
tracker) introduces a real feedback loop; if captures are silently dropping,
the usage-signal metrics will show it. For Phase 3, log-and-continue is
right.

**Observability:** emit a `console.error` for now. A future ADR can add
metrics when we have somewhere to send them (there is no metric system
today).

### D8 — Ship `backlog_recall` tool in Phase 3, not Phase 4

ADR 0092's Phase 3 sketch stopped at capture. But captures without a way
to read them back is dead code — and the whole point of Answer B above is
that episodic records should be *recallable independently* of full-task
search.

Add a thin MCP tool **as part of Phase 3** (not Phase 4):

```ts
backlog_recall({
  query: string,
  context?: string,        // scope to parent_id (e.g., FLDR-0001)
  tags?: string[],
  limit?: number,          // default 10
  layers?: MemoryLayer[],  // default: ['episodic']
})
→ MemoryResult[]
```

Thin: delegates to `ctx.memoryComposer.recall(query)` and returns the
results. No new core logic.

**Why bundle it:** without recall, there's no way to test Phase 3 end-to-end
as a user. Shipping capture without recall is like shipping a write-only
filesystem.

**What it gives us:** a clean Phase 4 entry point. Echo tracking (Phase 4)
hooks `backlog_recall` the same way it hooks `backlog_search`.

### D9 — Block on `TASK-0676`? No.

Phase 3 does not depend on Orama search being correct. Capture writes into
`InMemoryStore`, which uses keyword-based scoring (`packages/memory/src/in-memory-store.ts:34-40`).
Recall tests verify that scoring works for small corpora independently.

**But we document the dependency for Phase 4.** If we later swap in
`OramaEpisodicStore`, `TASK-0676` must be fixed first — otherwise memory
recall inherits the same ranking bugs.

### D10 — Instrumentation for Phase 4 and 5

While extending `WriteContext` and wiring capture, pre-plant two small hooks
so that Phase 4 and Phase 5 don't need to re-plumb:

1. **`metadata.usageCount: 0`** on every captured memory. Phase 4's echo
   tracker updates this field on recall-and-follow-up. Adding the field
   now means no schema migration later (everything in `InMemoryStore` is
   just an object).
2. **`expiresAt?: number`** on every captured memory — left undefined in
   Phase 3. Phase 5 (temporal facts) uses it. `MemoryEntry` already has
   this field; we just need the capture path to respect it (infrastructure
   only, no rules yet).

Both are zero-cost today and save refactoring later.

## Non-goals (explicit)

- **No LLM in the write path.** Reaffirming ADR 0092. If we find the digest
  is too thin, the fix is a richer deterministic rule, not an LLM call.
- **No cross-session durability.** Phase 3 memories die with the process.
- **No semantic/procedural layers.** Phase 3 writes only `episodic`.
- **No automatic GC.** Memories accumulate until shutdown. Forget-policy
  is a separate ADR.
- **No `backlog_forget` MCP tool.** Also a separate ADR. `MemoryStore.forget`
  exists; we just don't expose it.
- **No D1 episodic storage.** Worker is stateless; memory stays Node-only.

## Rollout

Phase 3 ships in two commits:

1. **3a — Capture + plumbing.** `WriteContext` extension, bootstrap wiring,
   capture rules, capture hooks in `create`/`update`, all tests. No new
   MCP tool yet. Merge target: backlog-mcp `main`, ships as 0.50.0.

2. **3b — Recall.** `core/recall.ts`, `backlog_recall` MCP tool,
   `backlog-mcp recall` CLI. Recall tests. Merge target: backlog-mcp
   `main`, ships as 0.50.1.

Two commits because 3a is purely internal (a new write side-effect); 3b is
user-visible (new tool to learn). Splitting lets us validate capture on real
usage before exposing recall.

## Open questions deferred (not blockers)

- **Memory compaction.** After N days, do old episodic memories merge into a
  semantic record? Needs Phase 4 echo data to do this well. Defer.
- **Cross-entity memory links.** If two artifacts reference the same
  concept, can their memories link? Maybe — but the backlog already links
  entities via `parent_id` and `references`. Don't add a parallel graph yet.
- **User-authored memories.** Is there a `backlog_remember` MCP tool for
  explicit capture? Probably — but not until we see how implicit capture
  performs. Defer.
