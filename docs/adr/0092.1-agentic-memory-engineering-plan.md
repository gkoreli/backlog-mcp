# 0092.1. Agentic Memory — Engineering Plan

**Status**: In Progress — **Phases 1 & 2 shipped 2026-05-07**; Phases 3–5 pending
**Target ADR**: 0092 (plugin-based agentic memory architecture)
**Locks**: Contract + file-level plan for Phases 1–5 of ADR 0092
**Defers**: Cross-backend memory sync, LLM-powered re-extraction, Karta graph upgrade

**Project folder:** [`FLDR-0001`](mcp://backlog/tasks/FLDR-0001.md) — all backlog-mcp work lives under this folder, enabling `backlog-mcp wakeup --scope FLDR-0001` for project-scoped briefings.

## Purpose

ADR 0092 decided **what** to build (native agentic memory on top of the existing
backlog as the source of truth, behind a `MemoryStore` plugin). This plan locks
**how** to build it — specifically, the four questions that the ADR left open
because answering them requires reading the current code, not deciding in principle.

After this plan, the five engineering phases become line-level work, not design
debate.

## The four open questions and their answers

### Q1 — Is `backlog_wakeup` a new tool, or a mode of `backlog_context`?

**Answer: new tool.** `backlog_context` requires a `task_id` or `query` as the
focal entity. Wake-up explicitly has **no focal** — it's "what should I know
right now, with no input." Overloading `backlog_context` with "if both args are
omitted, do something completely different" violates the single-responsibility
invariant the tool has today.

- `backlog_wakeup` lives at `packages/server/src/tools/backlog-wakeup.ts`
- Composes the same underlying stages as `backlog_context` but assembles a
  *time-oriented* payload rather than an *entity-oriented* one.
- Response layers mirror MemPalace's L0–L3, adapted to our substrates:

  | Layer     | Content                                                                  | Tokens (target) |
  |-----------|--------------------------------------------------------------------------|-----------------|
  | L0 identity | Static — from `.backlog/identity.md` if present, else project name only | ~50             |
  | L1 now      | Active tasks (status=in_progress, blocked) + current epic goals         | ~300            |
  | L2 recent   | Last 5 completions (evidence snippets), last 3 activity entries         | ~250            |
  | L3 on demand | Not in wake-up output. Caller follows up with `backlog_search` / `backlog_context` | 0               |

  Total: ~600 tokens, matching the ADR 0092 target.

### Q2 — Where does implicit memory capture live?

**Answer: as `layer: 'episodic'` `MemoryEntry` records written by a hook on
`BacklogService.update()` when the transition is `→ done`, and on
`BacklogService.create()` when `type === 'artifact'`.** No new Orama collection.

Rationale:
- `MemoryLayer = 'session' | 'episodic' | 'semantic' | 'procedural'` is already
  the taxonomy in `packages/memory/src/types.ts`. A task-done completion with
  its evidence is textbook *episodic* memory ("what happened, with what
  outcome") — the layer is named for exactly this.
- `MemoryComposer` already routes by layer. Registering an Orama-backed
  `EpisodicStore` is mechanical: `composer.register('episodic', oramaStore)`.
- **The memory entry is a pointer, not a copy.** `content` = the short
  human-meaningful line (task title + first evidence bullet, ~200 chars).
  `metadata.entity_id` = `TASK-0629`. On recall, the caller resolves the full
  task via `backlog_get(metadata.entity_id)` — the backlog remains the source of
  truth. We don't duplicate evidence into the memory store.
- **No LLM on the write path.** The agent wrote structured evidence; we index it
  verbatim. This is the ADR 0092 constraint, not a new decision.

Hook placement — see §"Phase 3 line-level plan" below.

### Q3 — Where does score decay go in the fusion pipeline?

**Answer: new `applyTemporalDecay` modifier, applied to the fused result before
coordination bonus.** One file, one function, one test.

Pipeline after this change:

```
BM25 hits  ──► minmaxNormalize ──┐
                                  ├─► linearFusion ─► applyTemporalDecay ─► applyCoordinationBonus ─► sorted
Vector hits ──► minmaxNormalize ──┘
```

**Why there and not inside `linearFusion`:**
- `linearFusion` is a pure score combiner with no time concept — adding decay
  inside it forces every future retriever (not just BM25+vector) to carry
  timestamps. Decay is orthogonal to fusion.
- Before coordination means decay shapes the *candidate order* the coordination
  bonus then preserves. Applying decay *after* coordination would mean a
  title-match bonus on a two-year-old task beats a body-only match on last
  week's work — the opposite of what "recency matters" means.
- **Half-life config, not `λ` config.** Humans reason about "how many days until
  this memory loses half its relevance", not about `λ` in the exponent. The
  function accepts `halfLifeDays`, converts internally: `λ = ln(2) / halfLifeDays`.
  Default: 30 days, tunable via composer config.

Signature (new, added to `packages/memory/src/search/scoring.ts`):

```ts
export function applyTemporalDecay(
  hits: ScoredHit[],
  getCreatedAt: (id: string) => number | undefined,
  opts: { halfLifeDays?: number; now?: number } = {},
): ScoredHit[];
```

Documents without a timestamp are left untouched (no decay applied — still
sorted by their pre-decay score). This keeps the decay opt-in per document:
ADRs, epics, and other timeless entities can skip the timestamp and keep their
rank.

### Q4 — How does echo/fizzle attribution work?

**Answer: infer from subsequent `backlog_get` / `backlog_context` calls in the
same session.** No new MCP tool, no explicit `ack`.

Contract:
- `backlog_search` returns results with their IDs.
- When the server sees a `backlog_get` or `backlog_context` call for an ID that
  appeared in a recent search result set (last 10 minutes, same session), that
  counts as an **echo** — usage recorded.
- Results that appeared in search results but never got a follow-up call by
  end-of-session **fizzle** — usage counter decayed.
- **Storage:** a compact ring buffer of `{search_id, result_ids, seen_ids,
  session_started_at}` kept in an `EchoTracker` inside the server, flushed to a
  `usageCount` field on entities periodically (every 100 searches or on
  shutdown).
- **Feedback into scoring:** `score *= 1 + log(1 + usageCount) * 0.1` applied
  inside `applyTemporalDecay`'s sibling, `applyUsageSignal`. Small multiplier
  (10%) — usage is a weak signal, decay is the stronger one.

Rationale for implicit over explicit `ack`:
- **No protocol change for clients.** Any MCP client already calls `get` /
  `context` on interesting results. No new tool to teach.
- **No burden on the agent.** The agent shouldn't have to remember to ack;
  its actual follow-up queries are the truthful signal.
- **Per-session bounded state.** `EchoTracker` doesn't need durable history —
  just enough to correlate within one agent's working loop.
- **Honest degradation.** If no agent follows up, `usageCount` stays 0 and the
  signal is inert — exactly correct.

**Deferred:** cross-session echo tracking and durable usage scores go on disk as
a separate JSON file after Phase 4 lands. Phase 4 itself is in-memory only.

## Phase-by-phase line-level plan

### Phase 1 — Score Decay  *(target: ~30 lines of code + 1 test file)* — **✅ shipped 2026-05-07**

**Files:**
- Add `applyTemporalDecay` to `packages/memory/src/search/scoring.ts`.
- Wire it into the fusion caller in `packages/memory/src/search/orama-search-service.ts`
  between `linearFusion` and `applyCoordinationBonus`.
- Pass entity `created_at` lookup via a closure provided by the search service
  (same pattern as `getText` / `getTitle` for coordination bonus).
- Test: `packages/server/src/__tests__/scoring.test.ts` — new describe block
  covering: (a) no-timestamp passthrough, (b) identical scores diverge after
  decay, (c) half-life sanity (30-day-old score is exactly half a same-day score
  given `halfLifeDays: 30`).

**Rollout:** ~~off by default behind `memory.scoring.halfLifeDays` in composer
config — unset means no decay, matches today's behavior.~~

**Actual rollout (2026-05-07):** Enabled by default with `halfLifeDays: 30` in
the singleton `BacklogService` constructor. Since this is a single-user tool,
gating with opt-in config added noise without value — the decision was to flip
on and measure. Docs without `created_at` pass through unchanged, preserving
backward compatibility for timeless entities (ADRs, schemas, etc.).

**What shipped:**
- `applyTemporalDecay()` and `DEFAULT_HALF_LIFE_DAYS = 30` exported from
  `packages/memory/src/search/scoring.ts`.
- `OramaSearchService` accepts `halfLifeDays` option; wires decay between
  `linearFusion` and `applyCoordinationBonus`.
- 10 new tests in `scoring.test.ts` covering the 30d half-life math, 60d
  quarter-life, disabled mode, future-date clamping, empty passthrough, and
  re-ranking semantics.

**Known follow-up:**
- [TASK-0676](mcp://backlog/tasks/TASK-0676.md) — **Search bug: exact-ID and
  title-substring matches buried behind weaker hits** — discovered during the

### Phase 2 — Wake-Up Tool  *(target: ~200 lines)* — **✅ shipped 2026-05-07**

**Files (as planned):**
- ~~New: `packages/server/src/context/wakeup-composer.ts`~~ → moved to
  `packages/server/src/core/wakeup.ts` (core placement per ADR 0090).
- New: `packages/server/src/tools/backlog-wakeup.ts` (MCP tool registration, thin).
- New: `packages/server/src/cli/commands/wakeup.ts` (CLI command, thin).
- Reuse: `context/stages/session-memory.ts` for L2's recent-activity slice.
- Register in `packages/server/src/tools/index.ts` and
  `packages/server/src/cli/index.ts`.

**Architectural correction mid-implementation:** Initially placed the composer
in `src/context/`, but per ADR 0090 the business logic belongs in `src/core/*`
with CLI + MCP as thin transport adapters. Moved to `core/wakeup.ts` with
signature `wakeup(service, params) → WakeupResult`. IO is injected
(`readIdentity`, `readOperations`) so core stays filesystem-free.

**Scope parameter design decision:** Added `scope?: string` to narrow the
briefing to a subtree. Uses plain `string` with runtime validation at the core
boundary (via `isValidEntityId` + substrate `isContainer` check), not a branded
`EntityId` type. Rationale captured in JSDoc on `WakeupParams.scope`:
- Matches codebase convention (all entity ID fields in `Entity` are plain `string`).
- Runtime validation is *stronger* than a brand — catches both malformed IDs
  AND non-container substrates (`task`, `artifact` are rejected; only
  `folder`/`milestone`/`epic` accepted).
- A wrong-typed brand would sail through; a wrong substrate cannot.

**Folder substrate as project abstraction:** Rather than add a new `Project`
substrate, wake-up uses the existing `Folder` substrate. A folder `FLDR-XXXX`
can hold epics, tasks, sub-folders (for artifacts or sub-projects), and any
other entity — `Folder` is the most flexible container. See `FLDR-0001`
(backlog-mcp itself) as the first production example.

**Tool signature:**

```ts
backlog_wakeup({
  max_tokens?: number = 800,       // hard cap; L0+L1+L2 truncated to fit
  include_completions?: boolean = true,
})
```

Response shape:
```ts
{
  identity?: string,              // L0 — nullable when no identity.md
  now: {
    active_tasks: EntitySummary[],
    current_epics: EntitySummary[],
  },
  recent: {
    completions: { id, title, evidence_snippet }[],
    activity: ActivityEntry[],
  },
  metadata: { token_count, generated_at }
}
```

**Identity file convention:** `$BACKLOG_DATA_DIR/identity.md` (plain markdown).
Absent → L0 omitted from response. We do **not** auto-create — the user writes
it when they want it.

**What shipped:**
- Core: `wakeup(service, params)` in `src/core/wakeup.ts` with `assertValidScope()`
  (throws `ValidationError` for malformed IDs or non-container substrates) and
  `descendantSet()` BFS walk using `service.list({ parent_id })`.
- MCP tool `backlog_wakeup` — snake_case params mapped to camelCase core params,
  `ValidationError` → `isError: true` response.
- CLI `backlog-mcp wakeup [--scope ID] [--max-completions N] [--max-activity N]
  [--evidence-chars N]` — pretty-prints L1/L2 sections with aligned columns.
- Types added to `src/core/types.ts`: `WakeupParams`, `WakeupResult`,
  `WakeupEntitySummary`, `WakeupCompletion`, `WakeupActivity` — all re-exported
  from `src/core/index.ts`.
- 19 new tests in `core-wakeup.test.ts` covering identity presence, L1
  filtering, L2 completions + evidence truncation, activity mapping, parent_id
  resolution (with `epic_id` alias fallback), and scope validation + descendant
  filtering (folder subtree, epic narrowing, empty folder, activity filter).

**Production usage validated (2026-05-07):**
- `backlog-mcp wakeup` returns full briefing from 663 tasks.
- `backlog-mcp wakeup --scope FLDR-0001` correctly narrows to backlog-mcp
  project work (2 active tasks, 1 epic, scoped activity).
- 663/663 tests pass, typecheck clean.

**Known gaps / follow-up:**
- L1 sections (active tasks, current epics) are **unbounded** — no
  `maxActiveTasks` / `maxCurrentEpics` caps. In a mature backlog with 19+
  active tasks, the briefing exceeds the ~600-token target. Add caps (default
  10 each) as a small follow-up before Phase 3 or leave unbounded and rely on
  `--scope` to narrow.
- No `TASK-0340`/`TASK-0596`/`TASK-0669` were under the project folder before
  this session — a one-time triage pass reparented them under `FLDR-0001`.
  See [`TASK-0676`](mcp://backlog/tasks/TASK-0676.md) for the search bug that
  made that triage harder than it should have been.

### Phase 3 — Implicit Memory Capture  *(target: ~100 lines + 2 test files)*

**Hook points:**

1. `BacklogService.update` / `D1BacklogService.update` — after the write
   succeeds, before returning, if `nextStatus === 'done' && prevStatus !== 'done'`:

   ```ts
   this.memoryComposer?.store({
     id: `mem-${entity.id}-${now}`,
     layer: 'episodic',
     content: `${entity.title} — ${entity.evidence?.[0] ?? ''}`.slice(0, 200),
     source: actor ?? 'system',
     context: entity.parent_id,
     tags: [entity.type, ...(entity.tags ?? [])],
     createdAt: now,
     metadata: { entity_id: entity.id, kind: 'completion' },
   });
   ```

2. `BacklogService.create` / `D1BacklogService.create` — after write, if
   `entity.type === 'artifact'`:

   ```ts
   this.memoryComposer?.store({
     id: `mem-${entity.id}-${now}`,
     layer: 'episodic',
     content: `${entity.title} — ${entity.description?.slice(0, 160) ?? ''}`,
     source: actor ?? 'system',
     context: entity.parent_id,
     tags: ['artifact', ...(entity.tags ?? [])],
     createdAt: now,
     metadata: { entity_id: entity.id, kind: 'artifact' },
   });
   ```

**Why inside the service (not a wrapper):** memory capture is a *consequence* of
a successful write. Wrapping the service means capture is separate from the
transaction and can drift (write succeeds, capture fails silently). Inside the
service, capture is fire-and-forget but co-located with the operation that
causes it — one place to audit, one place to test.

**Composer is optional:** both services accept the composer via constructor
(nullable). Missing composer → capture is skipped. The services still work
standalone (useful for tests and for the D1 edge build where we might ship
without memory on day one).

**Episodic store:** registered in the server bootstrap:
`composer.register('episodic', new InMemoryStore())` initially. A proper
`OramaEpisodicStore` (indexed for recall) can replace it later without
changing the hook contract.

**Test coverage:**
- `backlog-service.test.ts` — new describe: "memory capture on done-transition",
  "memory capture on artifact-create", "no capture when composer absent".
- A D1 parallel in `d1-backlog-service.test.ts`.

### Phase 4 — Echo / Fizzle  *(target: ~80 lines)*

**New file:** `packages/server/src/memory/echo-tracker.ts`
- Ring buffer of ~100 recent search emissions, keyed by a short `search_id`
  returned to the caller (existing `backlog_search` tool gains an opaque
  `search_id` field in its response).
- On `backlog_get(id)` and `backlog_context(task_id)`, the server looks up
  `id` in the buffer and, on hit, increments a per-entity `usageCount`.
- Every 100 searches (or on graceful shutdown), flush `usageCount` to the
  entity's memory metadata (`metadata.usageCount`).

**Scoring wire-up:** add sibling function `applyUsageSignal(hits, getUsage)` to
`scoring.ts`, applied after `applyTemporalDecay`. Coefficient 0.1 (see Q4 above).

**Test coverage:** a dedicated `echo-tracker.test.ts` — covers buffer eviction,
session-boundary behavior, and the fizzle semantics (no follow-up → no increment).

### Phase 5 — Temporal Facts  *(target: ~60 lines)*

**Schema extension:**
- `valid_from?: string` and `valid_until?: string` added to the shared entity
  shape in `packages/shared/src/schemas/*` (keep optional — backward-compatible).
- Search service filters out entities where `valid_until < now` unless the
  query explicitly opts in (`include_expired: true`).
- Wakeup's L1 skips expired entities.

**Migration:** none — both fields are optional and missing = "always valid",
matching today's behavior.

## Decisions NOT revisited by this plan

ADR 0092 already settled these — do not re-open:
- Native Orama implementation is the default store.
- `MemoryStore` is the plugin boundary.
- No LLM on the write path.
- No separate Python / HTTP services — TypeScript only.

## Rollout order and size

| Phase | Size       | Depends on | User-visible effect                                       |
|-------|------------|------------|-----------------------------------------------------------|
| 1     | ~30 LoC    | —          | Recent items rank above old items in `backlog_search`.    |
| 2     | ~200 LoC   | —          | New `backlog_wakeup` MCP tool returns ~600-tok briefing.  |
| 3     | ~100 LoC   | —          | Done tasks become recallable episodic memories.           |
| 4     | ~80 LoC    | 1 + 3      | Agent-used results rank higher over time.                 |
| 5     | ~60 LoC    | —          | Expired facts drop out of search.                         |

Phases are independently shippable. 1, 2, and 5 have no dependencies between
each other — any order is fine. 4 depends on 1 (to be the ranking pipeline it
plugs into) and 3 (to have a `usageCount` surface to write to). Recommended
order: **1 → 2 → 3 → 4 → 5** as in ADR 0092.

## Open questions deferred to a future ADR

- Cross-session / durable echo-fizzle tracking (Phase 4+).
- Orama-backed `EpisodicStore` (currently `InMemoryStore` placeholder in Phase 3).
- Compaction / forgetting policy for old episodic memories (GC, not decay).
- `backlog_forget` MCP tool — the write side of forget.

These are real, but don't block the five-phase rollout above.
