# Engineering Plan: Cron Entity Type (P0 for ARTF-0189 Co-Review)

**Status**: Ready to implement
**Target ADR**: 0097 (agentic storage engine) — Extension 1 only
**Blocks**: ARTF-0189 Phase 0 (Queue Awareness) and Phase 4 (Post-Merge Intelligence)
**Defers**: ADR 0097 Extensions 2 (aggregate endpoint) and 3 (composable markdown docs). Not in scope.

## Goal

Ship the cron entity type end-to-end so that this command works:

```
backlog_create type="cron" title="Review queue poll" \
  schedule="*/30 * * * *" \
  command="studio-agents check-reviews" \
  parent_id="EPIC-0043"
```

…and the resulting `CRON-0001` entity is:
- Stored as markdown with YAML frontmatter (`schedule`, `command`, `enabled`, `last_run`, `next_run` in frontmatter)
- Listable via `backlog_list type=cron`
- Editable via `backlog_update id="CRON-0001" schedule="*/15 * * * *" enabled=false`
- Updateable by external schedulers writing `last_run` / `next_run` (same `backlog_update` tool)
- Rendered in the viewer with the same DocumentView + MetadataCard as every other entity
- Filterable via `?type=cron` in the URL
- Indexed by search (title, description, references — same as every other entity)

**Explicit non-goals** (tracked in ARTF-0189, implemented elsewhere):
- The scheduler process (`studio-agents schedule`). This plan makes the *store* cron-aware. The scheduler is a separate, external MCP client — tracked as a follow-up task.
- Plugin discovery, sandboxing, stdout parsing, dedup. Scheduler concerns.
- Failure-artifact creation on cron tick errors. Scheduler writes these as normal artifacts.
- Aggregate endpoint or composable markdown dashboards. ADR 0097 Extensions 2-3, deferred.

## Design contract

### Frontmatter shape

A cron entity on disk (`$BACKLOG_DATA_DIR/tasks/CRON-0001.md`):

```markdown
---
id: CRON-0001
title: Review queue poll
status: open
type: cron
parent_id: EPIC-0043
schedule: "*/30 * * * *"
command: studio-agents check-reviews
enabled: true
last_run: "2026-04-28T22:00:00.000Z"
next_run: "2026-04-28T22:30:00.000Z"
created_at: "2026-04-28T19:33:53.671Z"
updated_at: "2026-04-28T22:00:00.000Z"
---

Optional markdown description — e.g. what this cron does, plugin expectations.
```

**Field semantics:**

| Field       | Type             | Who writes it           | Purpose                                               |
|-------------|------------------|-------------------------|-------------------------------------------------------|
| `schedule`  | `string`         | human/agent via `backlog_create` or `backlog_update` | 5-field cron expression, validated at write time |
| `command`   | `string`         | human/agent via `backlog_create` or `backlog_update` | Scheduler-interpreted invocation                 |
| `enabled`   | `boolean`        | human/agent via `backlog_update`                     | Whether the external scheduler should tick it    |
| `last_run`  | `string` (ISO-8601) | **scheduler only** via `backlog_update`         | Most recent tick timestamp                       |
| `next_run`  | `string` (ISO-8601) | **scheduler only** via `backlog_update`         | Next scheduled tick                              |

**Why `enabled` is separate from `status`** (reversing ADR 0096's original proposal):
`status` keeps its universal meaning (`open` / `in_progress` / `blocked` / `done` / `cancelled`).
`enabled: false` tells the scheduler to skip ticking without changing the semantic state of
the entity. A cron can be `status=open, enabled=false` (I care about it, but pause the ticking)
or `status=cancelled, enabled=false` (don't care, don't tick). Overloading `status` would
collide these cases.

### Validation rules

1. **Type guard at `backlog_create`**: `schedule`, `command`, `enabled` are only allowed when `type=cron`. Silently dropping them on other types is worse than rejecting — reject with a clear error.
2. **Type guard at `backlog_update`**: same — reject `schedule`/`command`/`enabled`/`last_run`/`next_run` edits if the target entity isn't type `cron`.
3. **`schedule` format validation**: must parse as a 5-field cron expression. Use a tiny parser (no heavy dep). Reject at write time with a message that points to the invalid field.
4. **`command`**: no server-side validation beyond "non-empty string." Interpretation is scheduler's problem.
5. **No autoincrement for `last_run`/`next_run`**: the server never sets them. They're blank until a scheduler writes them.

### Touch point 2: `packages/server/src/storage/schema.ts`

Additions to `CreateTaskInput` and `createTask()`:

```typescript
export interface CreateTaskInput {
  // ...existing...
  schedule?: string;
  command?: string;
  enabled?: boolean;
}

export function createTask(input: CreateTaskInput): Entity {
  // ...existing...
  if (input.schedule) task.schedule = input.schedule;
  if (input.command) task.command = input.command;
  if (input.enabled !== undefined) task.enabled = input.enabled;
  return task;
}
```

Default for `enabled` when not supplied: `undefined` (absent from frontmatter). Schedulers should treat "absent" as "not enabled" to be safe. If we want a default of `true` when `type=cron` and `enabled` is omitted, we set it in `createItem` (core layer), not the schema factory.

**Decision: default `enabled=true` on cron creation.** Rationale: if you're creating a cron, you almost always want it to start running. If you want to stage it disabled, you pass `enabled=false` explicitly. This matches cron-tab UX — you write a line, it runs. Document the default in the tool description.

LOC: ~7 lines.

`createItem` in `core/create.ts`:

```typescript
export async function createItem(service, params) {
  const { title, description, type, epic_id, parent_id, references,
          schedule, command, enabled } = params;

  // Validation: cron-only fields
  if ((schedule || command || enabled !== undefined) && type !== EntityType.Cron) {
    throw new ValidationError(
      'schedule, command, and enabled are only valid when type=cron'
    );
  }

  // Validation: cron requires schedule and command
  if (type === EntityType.Cron) {
    if (!schedule) throw new ValidationError('cron requires a schedule field');
    if (!command) throw new ValidationError('cron requires a command field');
    if (!isValidCronExpression(schedule)) {
      throw new ValidationError(`Invalid cron expression: ${schedule}`);
    }
  }

  // Default: cron starts enabled
  const resolvedEnabled = type === EntityType.Cron
    ? (enabled ?? true)
    : undefined;

  // ...existing id/createTask/add...
  const task = createTask({
    id, title, description, type, parent_id: resolvedParent, references,
    schedule, command, enabled: resolvedEnabled,
  });
  // ...existing...
}
```

`updateItem` in `core/update.ts`:

```typescript
export async function updateItem(service, params) {
  const { id, epic_id, parent_id, due_date, content_type,
          schedule, command, enabled, last_run, next_run,
          ...updates } = params;

  const task = await service.get(id);
  if (!task) throw new NotFoundError(id);

  // Cron-only field guard
  const isCron = task.type === EntityType.Cron;
  const hasCronFields = schedule !== undefined || command !== undefined
                     || enabled !== undefined || last_run !== undefined
                     || next_run !== undefined;
  if (hasCronFields && !isCron) {
    throw new ValidationError(
      'schedule, command, enabled, last_run, next_run are only valid on cron entities'
    );
  }

  // Assignments
  if (schedule !== undefined) {
    if (!isValidCronExpression(schedule)) {
      throw new ValidationError(`Invalid cron expression: ${schedule}`);
    }
    task.schedule = schedule;
  }
  if (command !== undefined) task.command = command;
  if (enabled !== undefined) task.enabled = enabled;

  // Nullable scheduler-owned fields
  if (last_run === null) delete task.last_run;
  else if (last_run !== undefined) task.last_run = last_run;
  if (next_run === null) delete task.next_run;
  else if (next_run !== undefined) task.next_run = next_run;

  // ...existing parent/epic/due_date/content_type logic...
  Object.assign(task, updates, { updated_at: new Date().toISOString() });
  await service.save(task);
  return { id };
}
```

**Cron expression validator**: write a small self-contained function in `packages/server/src/util/cron.ts` (~50 LOC). Validates 5-field format (`min hour dom month dow`), each field is `*`, a number in range, a list (`1,2,3`), a range (`1-5`), or a step (`*/5`). No external dep — simple regex + bounds check. This is well within the "small, self-contained" bar the codebase prefers.

LOC: ~80 lines of logic + ~50 lines for the cron validator.

## Risks and open questions

### Risks

1. **`ID_PATTERN` regex change is load-bearing**. Many places in the codebase depend on valid entity IDs. The regex extension from `MLST` to `MLST|CRON` is backward compatible for *read* — old entities still match — but we need to confirm there are no literal string checks for "MLST" as the last enum value anywhere. Mitigation: grep for `MLST` across the monorepo before commit; all references should be either the prefix constant or the enum, not a hardcoded last-element assumption.

2. **Cron validator correctness**. A hand-rolled cron parser has corner cases (day-of-week vs day-of-month conflict, month names `JAN-DEC`, dow names `MON-FRI`, `@every` shortcuts). Scope decision: **support only the classic 5-field numeric form (plus `*`, `,`, `-`, `/`)**. Reject everything else at validation time with a clear message. Schedulers that want richer syntax can validate independently on the tick side. This keeps the server-side validator small and testable.

3. **`enabled` default**. Defaulting to `true` on creation is a UX choice. If a user creates a cron and walks away, it will start firing as soon as a scheduler picks it up. Mitigation: the external scheduler must exist and be running; backlog-mcp by itself creates no side effects. Still, document clearly in the tool description.

4. **Viewer smoke test depends on backend registry being hot-reloadable**. Standard `pnpm dev` covers this. No risk beyond "remember to actually run it."

### Open questions

1. **Should `last_run`/`next_run` be first-class or folded into a nested `scheduler_state` object?**
   - First-class is simpler; matches the pattern of `due_date`, `content_type`, etc.
   - Nested would be cleaner if we add more scheduler state later (retry count, last error, etc.)
   - **Decision: first-class for now.** If we add ≥3 more scheduler fields we'll revisit. YAGNI wins.

2. **Should the scheduler get its own actor identity in the operation log?**
   - Today the operation log attributes writes to the MCP client (tool name + params). A scheduler writing `last_run` will show up as "backlog_update" by whatever actor is configured in the scheduler's MCP client.
   - This is *correct* but may look noisy when a scheduler is ticking dozens of crons. Mitigation is filter-side (add an actor filter to the activity panel), not core.
   - **Decision: ship as-is.** The cleanness of "scheduler is just another MCP client" outweighs the log noise; noise can be addressed by a follow-up to ActivityPanel filtering.

3. **Should we ship the humanized schedule display** (`*/30 * * * *` → "every 30 minutes") in v1?
   - Nice-to-have, but adds a library or a second parser.
   - **Decision: defer.** Raw cron expression is acceptable in the MetadataCard. If it becomes a pain, follow-up polish task.

4. **Should the scheduler live in this repo** (`packages/scheduler`) or in `studio-agents`?
   - Out of scope for this plan — but the plan assumes it is *external*, not in `packages/server`. That's the architectural commitment from ADR 0097.
   - **Recommended: `studio-agents schedule` subcommand.** One binary to install alongside backlog-mcp, keeps the store pure.

## Follow-ups (explicitly out of scope)

Track these as separate backlog tasks under EPIC-0002 (or EPIC-0043, whichever is the right container):
