# 0092.9. Agentic Memory Phase E — Usage Feedback: Research, Rulings, Plan

**Status**: Proposed
**Research method**: source-level review (repo clones at HEAD 2026-06-10) of
Mem0, Letta, Graphiti, Hindsight, Generative Agents + the credit-assignment
paper literature, focused on two questions: how do real systems attribute
"this memory was useful," and how do they handle **reads causing writes**.

## Research findings (cited, source-level)

### How the field tracks usage — nobody dirties the record on read

- **Mem0**: decay is *platform-only* — OSS only plumbs a `{"decay": bool}`
  toggle (`mem0/client/project.py:401-460`). Platform mechanics
  ([blog](https://mem0.ai/blog/introducing-memory-decay-in-mem0)): last-20
  access timestamps, **1.5×→0.3× bounded multiplier** that "reorders but
  never hides," reinforcement **fire-and-forget on a bounded executor**,
  candidate pool widened before truncation. Their OSS **OpenMemory** uses an
  **append-only sidecar table** `MemoryAccessLog(memory_id, accessed_at,
  access_type)` (`openmemory/api/app/models.py:176-188`), aggregated at query
  time — the memory row is never dirtied by a read.
- **Hindsight**: `access_count` column exists with a DESC index
  (`alembic/…initial_schema.py:279,337`), incremented via an **async batched
  task queue** (`engine/task_backend.py:241` — `access_count_update` with
  batched node_ids) — and is **not used in OSS ranking**. Their reranker
  bounds recency to **±10%** (`reranking.py`: `recency_alpha = 0.2`) and adds
  a `proof_count_boost` — corroboration-by-supporting-facts in ranking.
- **Letta**: archival memory has **zero usage tracking**; the only
  `last_accessed_at` is the file-block LRU (`orm/files_agents.py:73`),
  updated via targeted `UPDATE … SET last_accessed_at=now()` without loading
  the row (`files_agents_manager.py:439`). Sleep-time agents decide rewrites
  by **prompt + turn counter** (`sleeptime_multi_agent_v2.py:114-119`) — no
  telemetry.
- **Generative Agents** (arXiv 2304.03442): the original decay-on-access —
  `recency = 0.995^(hours since last retrieval)`, `last_accessed ← now` for
  every retrieved node (`retrieve.py:new_retrieve()`) — but `last_accessed`
  is **never persisted**: `AssociativeMemory.save()` omits it; it resets on
  reload. The canonical design treated access state as ephemeral.
- **Graphiti**: no usage tracking; freshness = bi-temporal invalidation
  (`edge_operations.py:resolve_edge_contradictions()`: `invalid_at` = when
  the fact stopped being true, `expired_at` = when the system learned it).

### Update rules from the literature

- **MemoryBank** (arXiv 2305.10250 §2.3): Ebbinghaus `R = e^(−t/S)`; on each
  recall `S += 1`, `t ← 0` — **usage extends the half-life** (spacing effect).
- **Fine-Mem** (arXiv 2601.08435 §3, Alg. 1): evidence-anchored credit —
  reward flows to memories **actually used as evidence**, diluted by
  `1/(|M_j|·n)`; mere retrieval is weak "participation credit." The
  RL-free lesson: *expansion/citation ≫ appearing in a result list*.
- **FSFM** (arXiv 2604.20300, production data): multi-factor retention with
  **protected classes and grace periods**; capacity-triggered eviction by
  composite score, never threshold deletion; "staircase" reinforcement with
  diminishing resets. Saturating counters everywhere else: Postgres clock-
  sweep caps usage_count at 5; Redis LFU uses an 8-bit Morris counter.

### The reads-cause-writes precedent: atime → relatime → lazytime

Filesystems solved our exact storage dilemma: strictatime (write inode per
read) was abandoned ecosystem-wide; **relatime** (default since kernel
2.6.30) writes atime only when it's older than mtime or **>24h stale**;
**lazytime** keeps it current in memory and flushes opportunistically
([LWN 621046](https://lwn.net/Articles/621046/),
[LWN 620192](https://lwn.net/Articles/620192/)). Bounded, condition-gated
persistence that keeps semantic consumers working — that is the design.

## Rulings (locked by this ADR)

**R-13. Storage is the relatime hybrid (Option C).** Raw usage events append
to a JSONL audit file in the data dir (`memory-usage.jsonl` — human-readable,
greppable, replayable, git-ignorable by choice); the human-visible summary
(`usage_count`, `last_used_at`) lives in frontmatter but is flushed
**relatime-gated**: only when the count crosses a bucket boundary
(1,2,3,5,8,13,21,34…) or the stored `last_used_at` is >24h stale. Per-read
frontmatter writes (strictatime) are rejected; invisible-only sidecars
(pure Option B) violate the visibility tenet; never-persisted state
(Generative Agents) evaporates on restart.

**R-14. Only strong events count.** `usage_count` counts **expand**
(`backlog_get` on a MEMO- id) and **cite** (MEMO- ids appearing in
subsequently written content) — Fine-Mem's performance credit. Appearing in
a stub list is logged to JSONL (it is the recall-demand signal Phase D
deferred) but does not bump the counter. Recall stays a pure read.

**R-15. Bounded multiplier with a grace period.** In memory recall ranking:
memories younger than 14 days are neutral (1.0 — they haven't had a chance
to earn usage); otherwise
`factor = 0.3 + 1.2 · min(usage_count, 8)/8 · 2^(−days_since_last_use/30)`,
clamped to [0.3, 1.5]. Applied after relevance scoring inside the memory
store — reorders, never filters; memory-only (the generic entity pipeline is
untouched). Saturating count (cap 8 effective, 255 stored).

**R-16. The JSONL is the recall-demand log.** Every recall appends one
batched line (query, returned ids, scores) — Hindsight's batched
`access_count_update` shape. This closes the open signal from ADR 0092.7
(consolidation ripeness by demand) for a future loop.

**Deferred to the next loop**: corroboration boost in live ranking
(Hindsight caps it at ±10% and warns boosts must not dominate weak base
signals — we want real usage data first); capacity-triggered archive GC with
protected classes (FSFM shape); staleness report. **Rejected**: LLM-judged
forgetting (Letta sleeptime prompts, A-MEM evolution, RL managers — no LLM
in the server); threshold deletion (MemoryBank, mem0 issue #5330 — soft
expiry only); unbounded usage influence.

## Engineering plan

| File | Change |
|---|---|
| `shared/src/substrates/memory.ts` | `last_used_at?: string` |
| `server/src/memory/usage-tracker.ts` **(new)** | `MemoryUsageTracker`: `recordRecall(query, ids)` (JSONL only), `recordExpand(id)`, `recordCitations(text)` (regex MEMO- ids); relatime-gated flush via injected service; injected JSONL appender (node-only IO stays at the edge) |
| `server/src/memory/usage-signal.ts` **(new)** | pure `usageFactor(memory, now)` per R-15 |
| `server/src/memory/backlog-memory-store.ts` | apply `usageFactor` to recall scores (after search, before sort/limit) |
| `server/src/memory/bootstrap.ts` | default tracker singleton wired to data dir |
| `tools/backlog-get.ts`, `cli/commands/get.ts` | expand hook: MEMO- id → `tracker.recordExpand` |
| `tools/backlog-recall.ts`, `cli/commands/recall.ts` | demand log hook |
| `core/remember.ts` (+ tool/CLI deps) | citation scan of content via tracker |
| tests | `usage-signal.test.ts` (factor table), `usage-tracker.test.ts` (gating buckets, JSONL lines), contract E2E (expand → flush → boosted rank) |
