Building the F3 fix (CLI get --context) I found the Tier-1 'expand' telemetry emission living inside MemoryUsageTracker.recordExpand — the same method that grants the ADR 0092.9 R-14 ranking credit (overlay line + counter bump). Two different concepts shared one name and one seam: a ranking signal (agent chose this memory after seeing the stub) and an observation event (agent performed the ADR 0114 neighborhood act). Granite's ruling forced the split: recordExpand keeps ranking credit only; recordContextExpand emits the Tier-1 event only, on entity-id gets with context=true from either surface. The lesson: when an observation event and a ranking signal share a method, they will eventually need different firing rules — keep telemetry seams separate from ranking seams from day one, and when told 'ranking untouched', grep for what else fires on the same call path before moving an event.
