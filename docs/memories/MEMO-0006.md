Fixing the review 0001 telemetry MEDIUMs I earned two lessons. (1) A long-lived component shared across stateless requests (LocalRuntime and its RetrievalTelemetry live for the server's lifetime; the /mcp transport builds a fresh server per request) cannot carry per-request context through ANY construction seam — constructor deps, resolver, tool deps all cross requests. Thread it through the async execution context instead: retrieval-telemetry.ts now holds the repo's first AsyncLocalStorage (withRequestTelemetrySession wraps connect+handleRequest in hono-app's /mcp), and telemetrySessionId reads env > request scope > process mint. The same shared-runtime/per-request shape is what the HIGH-1 request-selected identity fix will face — the pattern now exists as prior art. (2) memfs is not POSIX-faithful for renameSync: renaming a file onto a NON-EMPTY DIRECTORY silently clobbers it where POSIX throws ENOTEMPTY/EISDIR — so under this repo's memfs law, failure-injection tests must inject at the fs API boundary (vi.spyOn on the mocked node:fs module), never via filesystem states that 'should' error. My rotation fail-open test asserted A-then-B survived an EACCES rename spy; the directory-occupation trick silently rotated and the test caught memfs, not my code.
